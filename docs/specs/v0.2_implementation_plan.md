## How to use this playbook (for the implementer)

- **The spec is authoritative.** This playbook references the spec's sections by name
  and adds the *executable detail*. Where they disagree, the spec wins.
- **Read the SHIPPED v0.1 source, not just the v0.1 spec.** v0.2 extends real code. The
  load-bearing seams (verified during planning):
  - `core/bootstrap.py::_CORE_ADAPTER_MODULES` ÔÇö add every new core adapter module here.
  - `core/mcp/server.py::build_server(config)` + `_FastMcpAdapter` (has `run()` for
    stdio) + `_core_registrars(context)` + `_V0_2_STUB_TOOLS` + `KNOWN_READ_ONLY_TOOLS`.
  - `core/mcp/context.py::QueryContext` ÔÇö the per-tool seam.
  - `core/config.py` ÔÇö TypedDicts + `PlaneRegistries` + `_resolve_dashboard_packs` (the
    deferred `git` selector already emits a `selector="git"` pack the v0.1 provider
    rejects; v0.2 makes provisioning it real).
  - `core/collector.py::Collector` ÔÇö the per-cycle loop with the proven per-source
    consecutive-failure counter the alert `for_cycles` debounce reuses.
  - `tests/unit/test_no_write_actions_guard.py` ÔÇö extend (don't rewrite) for the
    path-scoped `sns.publish` allowlist.
  - `docker-compose.yml` (v0.1) ÔÇö the LOCAL path; do NOT delete it. The Helm chart is the
    HOSTED analogue; the same GHCR image runs in both.
- **TDD-first across all applicable layers.** Red tests first, then implement to green.
  For Terraform the red-first artifact is the plan-assertion test + `terraform validate`;
  for Helm it is the render-assertion test + `helm lint` + `helm template | kubeconform`.
- **One semantic commit per phase.**
- **The v0.1 gate is the base gate for every phase** (extended per phase). Use the
  Python 3.12 venv by absolute path: `.venv/bin/ruff`, `.venv/bin/mypy`,
  `.venv/bin/pytest` (the shell default `python` is conda 3.11 and cannot parse PEP 695).

## 5. Definition of Ready for `fb`

### Are the spec + this playbook sufficient for autonomous implementation? **YES.**

The spec provides the *what* (new adapters, tools, HTTP transport, alert model, the EKS
Terraform surface + IRSA + ingress, the Helm chart, the auth model, DoD, tests). This
playbook provides the *executable how* on the REAL shipped seams, the red-test-first
sequence (incl. the hermetic terraform/helm gates), the phase DAG, exit gates, and a
resolved risk register. The six hosting choices are FIRM operator decisions (no open
forks).

```

[x] Brand-neutrality enforced across modules/ + deploy/ + charts/
[x] The six hosting decisions are FIRM (operator-confirmed) ÔÇö no defaults left to choose
```

### External prerequisites the OPERATOR provides (NOT blockers for `fb`'s build/CI)

Needed only for the live hosted `terraform apply` + `helm install` + GitHub login ÔÇö not
for `fb` to build, unit-test, or pass CI (CI is hermetic):

1. A **Panoptes home AWS account** + region + a **dedicated VPC** (SAME account as the
   observed infra ÔÇö decision #1; never an observed VPC).
2. A **GitHub OAuth app**: client id/secret + the **org (and/or team) allowlist** that
   gates access (decision #5).
3. A **`panoptes.<domain>`** hostname (Route53 zone or a manual A record); TLS is issued
   by cert-manager + Let's Encrypt (decision #3 ÔÇö no ACM cert needed).
4. The per-env **`PanoptesReadRole` ARNs** (consumer's IaC, **in-account** ÔÇö empty list ok
   for dev-only).
5. An **SNS topic ARN** + a **Slack webhook URL** for alerts.
6. A read-only **in-cluster ClusterRole/SA** (Panoptes' own cluster, self-observe) + a
   per-env **kubeconfig/SA token** for any observed cluster the `kubernetes` source reads.

### Verdict

**`fb`-ready. No blockers.** Begin at Phase 0 and proceed phase-by-phase, gating each on
