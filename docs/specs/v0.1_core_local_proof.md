## Product Specification

### Overview

Panoptes is a standalone, pluggable normalizing meta-layer over the monitoring
tools an organization already runs. v0.1 proves the core abstraction against one
live environment at **zero cloud cost**: a local `docker-compose` stack
(VictoriaMetrics + Grafana + a Python collector + an MCP server) pulls read-only
from the reference consumer's `dev` environment (CloudWatch + Sentry + an HTTP
`/health` endpoint), folds those signals into one canonical model, stores
metrics in VictoriaMetrics, and serves the same normalized view through **two
faces that never drift**: Grafana dashboards and an MCP (stdio) server.

v0.1 deliberately ships **two faces, one store** and the **four plug-planes**
(Source / Store / Notifier / Dashboard) as real, typed, registry-driven
abstractions — so adding a tool later is "one adapter class + one config block,"
not a refactor.

### Strategic positioning — the core/consumer-pack boundary (load-bearing)

The single most important architectural invariant in v0.1:

> **`core` knows NOTHING about any consumer.** Everything domain-specific lives
> in a **consumer pack** — dashboard JSON, optional custom adapters, and the MCP
> tools they register — that **lives in the consumer's own repo** and is
> **injected into Panoptes at deploy time** via a config pointer, never bundled
> into this repo.

Two dashboard tiers make this concrete:

- **Tier 1 — core packs** ship inside Panoptes under `core/dashboards/`
  (errors-sentry, logs, overview in v0.1) and are always provisioned.
- **Tier 2 — consumer packs** are NOT stored in Panoptes. They live in the
  consumer repo (e.g. `<consumer>/ops/panoptes/dashboards/<name>/dashboard.json`)
  and are injected at deploy time: a **mounted read-only dir** locally
  (`${CONSUMER_PACK_DIR:?}:/packs/consumer:ro`), or a **pinned git repo+ref+subdir**
  when hosted (v0.2). The Grafana provider globs `dashboards/**/dashboard.json`
  across the core dir + the injected consumer dir and provisions all of them. The
  injected dir also carries the consumer's optional `pack.py` (custom adapters +
  MCP tools).

To **prove the injection path** in v0.1 without baking any domain content into
Panoptes, the in-repo `examples/` directory ships a **generic, brand-neutral demo
pack** (a `demo` dashboard + a synthetic MCP tool) plus an integration-test
fixture. A real consumer (e.g. the reference game platform with its
matchmaking/allocator pack) would instead point the injection config at its own
repo dir — that content is documented but not built here.

This is enforced on three levels:

1. **Structural import discipline (the primary anti-coupling control):** no module
   under `core/` may contain `from examples` / `import examples`, and no `core/`
   module may resolve a path inside `examples/`. The CI guard greps `core/**/*.py`
   for this and fails on any match.
2. **Generic consumer-domain term check (defense in depth):** the same guard fails
   if a `core/` module mentions generic consumer-domain identifiers (`allocator`,
   `matchmaking`, `agones`). No brand token is embedded in the guard — the repo is
   brand-neutral, so the guard needs no brand literal. (If a future deployment ever
   needs to tripwire a specific brand string, it would be stored byte-encoded and
   decoded at test runtime so the plaintext never appears in source.)
3. **Runtime additive-injection invariant:** a test loads the demo pack via the
   `PANOPTES_CONSUMER_PACK` hook and asserts the injection is **purely additive and
   reversible** — core's own registrations are unchanged, the consumer's adapter/tool
   appears only as an addition, and re-running with the hook unset yields a core-only
   registry identical to the no-pack baseline (see § Tests → Guards).

The demo pack is the *fixture that exercises the injection path*, never a special
case baked into core.

### User journey (v0.1)

1. An operator clones the repo, copies `examples/demo-pack/.env.example` to
   `.env`, fills in the dev AWS profile + Sentry token + health URL, and points
   `CONSUMER_PACK_DIR` at the demo pack dir (or their own consumer repo's pack
   dir).
2. `docker-compose up` starts VictoriaMetrics, Grafana (auto-provisioned with the
   core packs + the injected consumer/demo pack), the collector, and the MCP
   server. The demo pack dir is bind-mounted read-only at `/packs/consumer`.
3. The collector assumes the read-only role (or uses the local dev profile),
   pulls signals from CloudWatch/Sentry/health on a schedule, normalizes them,
   and writes metrics to VictoriaMetrics.
4. The operator opens Grafana → sees core Errors/Logs/Overview + the injected
   `demo` dashboard, switchable via the `env` template variable.
5. An LLM operator registers the **stdio** MCP server in their client and asks
   `describe_health(env="dev")`, `search_incidents(...)`, and the demo pack's
   synthetic tool — getting the *same data* the dashboards show.

## Data Model — the canonical signal model

Greenfield: all types are new. No migration. Precise typing throughout — no
untyped escape hatches (no `Any`, no bare `dict`). Use `TypedDict`, `dataclass`,
`enum.Enum`, and `Protocol`.

### Signal kinds

```python
class SignalKind(enum.Enum):
    METRIC = "metric"
    LOG = "log"
    TRACE = "trace"       # contract present; no v0.1 source provides it
    INCIDENT = "incident"
```

### Core types (`core/model.py`)

| Type | Kind | Key fields |
|------|------|-----------|
| `MetricSignal` | metric | `name: str`, `value: float`, `timestamp: datetime`, `labels: Labels` |
| `LogSignal` | log | `timestamp: datetime`, `message: str`, `level: LogLevel`, `labels: Labels` |
| `IncidentSignal` | incident | `id: str`, `title: str`, `level: IncidentLevel`, `first_seen: datetime`, `last_seen: datetime`, `count: int`, `labels: Labels` |
| `TraceSignal` | trace | `trace_id: str`, `spans: list[Span]`, `duration_ms: float`, `labels: Labels` (declared; not produced in v0.1) |
| `Span` | — | `name: str`, `start: datetime`, `duration_ms: float`, `parent_id: str \| None` (element type of `TraceSignal.spans`) |

- `Labels = dict[str, str]` (a `TypedDict`-free alias is acceptable here because
  label keys are open-ended; values are always `str`). **`env` is mandatory** on
  every signal — enforced in a `__post_init__` validator that raises if absent.
- `CanonicalSignal = MetricSignal | LogSignal | IncidentSignal | TraceSignal`
  (a typed union, discriminated by an explicit `kind: SignalKind` field on each).
- `LogLevel` / `IncidentLevel` are enums (`DEBUG…CRITICAL` / `INFO…FATAL`).

### Derived metrics (the parity bridge)

Logs and incidents are **queried live** from their source by default
(ARCHITECTURE §2), but to render rate panels in Grafana from the store, sources
MAY emit derived gauge metrics alongside their native signals:

| Derived metric | Source | Labels |
|----------------|--------|--------|
| `panoptes_sentry_incident_count` | sentry | `env`, `level`, `project` |
| `panoptes_log_error_rate` | cloudwatch (logs) | `env`, `log_group` |
| `panoptes_health_up` (1/0) | http-health | `env`, `url` |
| `panoptes_health_latency_ms` | http-health | `env`, `url` |

`ASSUMPTION:` derived metrics are named with a `panoptes_` prefix to avoid
colliding with native upstream metric names in PromQL.

### Query / aggregation types

| Type | Purpose |
|------|---------|
| `TimeWindow` | `start: datetime`, `end: datetime`; helper `last(minutes=…)` |
| `MetricQuery` | `expr: str` (PromQL), `window: TimeWindow`, `step_seconds: int` |
| `MetricSeries` | `metric: str`, `labels: Labels`, `points: list[tuple[datetime, float]]` |
| `SourceHealth` | `reachable: bool`, `detail: str`, `checked_at: datetime` |
| `Alert` | `name: str`, `severity: str`, `message: str`, `labels: Labels` |
| `DashboardPack` | `id: str`, `tier: Literal["core","consumer"]`, `json_path: Path` (core packs resolve under `core/dashboards/`; consumer packs resolve under the injected `dashboards.consumer_pack.path`) |

### Storage shape (VictoriaMetrics)

Metrics are written via the VictoriaMetrics **Prometheus remote-write** or the
JSON line import endpoint (see § API Surface). Every series carries `env` as a
label plus source-specific labels. Logs/incidents are NOT persisted in v0.1
(queried live); only their derived gauge metrics are stored.

## API Surface

### Plug-plane Protocols (`core/planes/`)

The four typed Protocols, finalized from the ADAPTERS.md sketches:

```python
class Source(Protocol):
    type: str
    def capabilities(self) -> set[SignalKind]: ...
    def fetch(self, window: TimeWindow) -> list[CanonicalSignal]: ...
    def health(self) -> SourceHealth: ...

class Store(Protocol):
    type: str
    def write(self, signals: list[CanonicalSignal]) -> None: ...
    def query(self, query: MetricQuery) -> list[MetricSeries]: ...

class Notifier(Protocol):
    type: str
    def notify(self, alert: Alert) -> None: ...

class DashboardProvider(Protocol):
    type: str
    def provision(self, packs: list[DashboardPack]) -> None: ...
```

### Registry (`core/registry.py`)

A type-safe registry per plane. Adapters self-register via a decorator keyed on
their `type` discriminator:

```python
SOURCES: Registry[Source] = Registry("source")

@SOURCES.register("cloudwatch")
class CloudWatchSource: ...
```

- `Registry[T].build(type: str, config: Mapping[str, str | int | bool | list[str]]) -> T`
  constructs an adapter from a config block; **fails fast** with
  `UnknownAdapterError(plane, type)` listing available types when `type` is not
  registered (ADAPTERS §3 "no adapter for type X").
- Generic over the plane Protocol so mypy enforces that a registered class
  satisfies the plane.
- Core registers only generic adapters. Consumer packs call
  `SOURCES.register(...)` from their own module, loaded via an entry-point /
  explicit import (see § Consumer-pack loading).

### Source adapters (core)

| Adapter | Upstream | Provides | Mechanism |
|---------|----------|----------|-----------|
| `cloudwatch` | CloudWatch metrics + Logs | `capabilities() == {METRIC, LOG}` (alarm→incident is v0.2) | `boto3` `GetMetricData`, `FilterLogEvents`; assume-role via STS or local profile |
| `sentry` | Sentry REST API | `capabilities() == {INCIDENT, METRIC}` — `incident` native + the derived `panoptes_sentry_incident_count` gauge metric | `httpx` against `/api/0/projects/{org}/{project}/issues/` filtered by `environment` |
| `http-health` | an HTTP `/health` endpoint | `metric` (`panoptes_health_up`, `panoptes_health_latency_ms`) | `httpx` GET + latency timing |

- All sources are **read-only w.r.t. observed systems** (IAM.md `PanoptesReadRole`,
  the single read-only credential domain). No `Put*`/`Create*`/`Write*` calls
  anywhere — enforced by a CI grep guard.
- AWS auth: `ASSUMPTION:` v0.1 uses the local dev AWS profile via a configurable
  `profile` / `assume_role_arn`; `assume_role_arn` takes precedence when set
  (forward-compatible with IAM.md `PanoptesReadRole`). The cloudwatch source
  config carries a first-class **`external_id: str | None`** field alongside
  `assume_role_arn` (interpolated from `${PANOPTES_EXTERNAL_ID}`): it is the value
  the cross-account trust policy requires (IAM.md §A confused-deputy guard), so it
  has a real home in the config contract rather than only prose. When
  `assume_role_arn` is set, `external_id` SHOULD be supplied to match the
  trust-policy `sts:ExternalId` condition; the assume-role call passes it through
  to STS. It is not an env-var name itself but a config field whose value is
  `${VAR}`-interpolated.

### Store adapters (core)

| Adapter | Behavior |
|---------|----------|
| `victoriametrics` (default) | writes via VM import endpoint; queries via PromQL `/api/v1/query_range`; returns `MetricSeries` |
| `passthrough` | no persistence; `write` is a no-op recording in-memory last-batch; `query` raises `CapabilityError` (a deliberately trivial store for tests / source-only runs) |

### Dashboard provider (core)

| Adapter | Behavior |
|---------|----------|
| `grafana` | Provisions BOTH tiers by globbing `dashboards/**/dashboard.json` across (a) the in-repo `core/dashboards/` (tier-1 core packs) and (b) the **injected consumer-pack dir** (tier-2, mounted read-only at `/packs/consumer`). Uses Grafana file-based provisioning + a thin HTTP client to verify dashboards loaded. The MCP server is **fully read-only**: there is NO `create_dashboard`/`save_dashboard` and no write path of any kind (see § MCP server). |

`ASSUMPTION:` v0.1 uses Grafana **file provisioning** (a provisioning YAML + the
globbed dashboard JSON dirs) rather than the Grafana Terraform provider — the
Terraform provider is v0.2 when the host is Terraform-managed. The
`DashboardProvider.provision(packs)` implementation receives both the resolved
core packs and the injected consumer pack (resolved from `dashboards.consumer_pack.path`),
writes/syncs their JSON into the mounted provisioning dir, and pings the Grafana
API to confirm load. New consumer dashboards are authored as code in the consumer
repo and injected — never created through MCP.

### Notifier adapter (core)

| Adapter | Behavior |
|---------|----------|
| `logging` | renders the `Alert` to a structured log line (f-string). The only v0.1 notifier; proves the plane. `sns`/`slack` deferred to v0.2. |

### MCP server (`core/mcp/` + consumer-pack tools)

FastMCP, **stdio transport** for v0.1 (HTTP + SSO/OIDC is v0.2). All query tools
read the shared store / live sources — never write to observed systems.

**Discovery / parity (core):**
- `describe_signal_catalog() -> SignalCatalog` — environments, configured
  sources + their capabilities, known metric names, dashboard ids.
- `list_dashboards() -> list[DashboardSummary]` — the dashboard catalog (core +
  loaded consumer packs).
- `get_dashboard_data(id: str, env: str) -> DashboardData` — per-panel title +
  PromQL target(s) + executed series from the store (see Open Question 3).

**Query (core):**
- `query_metric(env: str, name: str, window: str, filters: Mapping[str, str] | None) -> list[MetricSeries]`
  — a thin PromQL passthrough to the store with caller-supplied query parameters.
  It is the only registered tool that reads the store directly with caller-supplied
  PromQL, so it is the natural surface for a `passthrough`-store `CapabilityError`
  (see § Tests → MCP).
- `search_incidents(env: str, window: str, tag: str | None, level: str | None) -> list[IncidentSignal]`
- `search_logs(env: str, query: str, window: str, level: str | None) -> list[LogSignal]` (env-first by convention — every tool leads with `env`)
- `describe_health(env: str) -> HealthRollup` — the "one thing to look at"
  rollup: per-source reachability + key derived metrics + open incident count.

**Consumer-pack-registered (injected demo pack, NOT core):**
- `get_demo_signal(env: str, window: str) -> DemoSignal` — a generic,
  brand-neutral synthetic tool registered by the demo pack's `pack.py` via the
  `register_tools(mcp_server)` hook. It reads `panoptes_*` metrics from the store
  and proves the consumer-tool registration path end to end. A real consumer pack
  (e.g. a platform's `get_allocator_pressure`) registers its own tools the same
  way, from its own repo (see Open Question 5).

**Contract (all tools) — fully read-only:**
- The MCP server is **fully read-only**: every tool is read-only w.r.t. observed
  systems AND there is **no write path of any kind** — no `create_dashboard`, no
  `save_dashboard`, no save-to-repo, no runtime mutation. New dashboards are
  authored as code in the consumer repo and injected at deploy time.
- Accept and respect an `env` argument (or `"all"`).
- **Fail explicitly** when a required source/capability isn't configured
  (`CapabilityError` surfaced as a structured MCP error) — never a silent empty
  result. Capability negotiation: a tool consults `describe_signal_catalog`
  internally and returns "no source for X" rather than guessing.
- Tool return types are precise `TypedDict`s (FastMCP serializes them) — no
  `Any`. Note: due to a known FastMCP / PEP-563 interaction, do **not** add
  `from __future__ import annotations` to FastMCP tool modules that return
  nested-TypedDict shapes (deferred annotations break FastMCP's schema
  generation for nested TypedDicts).

**v0.1 contract for config-listed-but-unimplemented tools (`compare_envs`,
`get_slo`, `get_cost`):** the `mcp.tools:` config block (ADAPTERS §3) may list
v0.2 tools. Unlike an unknown *adapter* `type` (which fails fast at config-resolve
time, because it names a non-existent plugin), these are **valid future tools**,
not unknown adapters — so listing them is **NOT a config-resolve failure**.
Instead, in v0.1 they are registered as **stub tools** that return a structured
`"not available in v0.1 (ships v0.2)"` MCP error **at call time**. This gives a
clear, testable behavior contract: a caller invoking `compare_envs(...)` over the
v0.1 server gets an explicit not-available MCP error, never a missing-attribute
crash and never a resolve-time rejection. (The v0.1 example config at
`examples/demo-pack/panoptes.yaml` lists only the v0.1-implemented tools and does
NOT include the v0.2 stubs — see § Configuration — so the default path exercises
only real tools; the stub contract is exercised by an explicit test that lists a
v0.2 tool.)

### Consumer-pack loading (injection, not bundling)

The consumer pack is **external** — it lives in the consumer's own repo and is
injected, never imported from inside Panoptes. Two halves:

- **Dashboards** are resolved from `dashboards.consumer_pack.path` (the mounted
  dir, `/packs/consumer` in compose) and globbed alongside the core packs by the
  Grafana provider.
- **Code** (optional custom adapters + MCP tools) lives in the injected dir's
  `pack.py`. `ASSUMPTION:` v0.1 loads it by an explicit `PANOPTES_CONSUMER_PACK`
  module path env var (default unset = core-only; the compose file sets it to the
  mounted demo pack's `pack.py`). On import the pack (a) registers any custom
  adapters via the registries and (b) registers its MCP tools via a
  `register_tools(mcp_server)` hook. This keeps core free of any consumer import.
  v0.2 may switch to Python entry-points and the `git` injection variant.

For v0.1 the injected pack is the in-repo **`examples/demo-pack/`** (a generic,
brand-neutral fixture), mounted via `CONSUMER_PACK_DIR`. It proves the path a real
consumer repo would use.

## Configuration

### Loader (`core/config.py`)

Parses the `panoptes:` YAML schema (ADAPTERS §3). Responsibilities:

1. Load YAML, validate top-level shape against `TypedDict` schemas
   (`PanoptesConfig`, `EnvironmentConfig`, `SourceConfig`, `StoreConfig`,
   `NotifierConfig`, `DashboardsConfig`, `SloConfig`, `McpConfig`).
   `DashboardsConfig` has `provider`, `env_variable`, `core_packs: list[str]`,
   and `consumer_pack: ConsumerPackConfig | None` where `ConsumerPackConfig` is a
   union of a `path` (mounted dir, v0.1) or a `git` (repo+ref+subdir, v0.2,
   parsed-but-deferred). The Grafana provider resolves core packs from
   `core/dashboards/` and the consumer pack from the injected dir.
2. **`${VAR}` interpolation** from the process environment. Fail fast with a
   clear message naming the missing variable (`MissingEnvVarError("DEV_HEALTH_URL")`).
   Never inline secrets in config (IAM.md / ADAPTERS §3 convention).
3. **Fail-fast on unknown adapter `type`** — delegates to the registry's
   `build`, which raises `UnknownAdapterError` listing valid types.
4. `enabled: false` environments parse but produce no live adapters (wired-but-inert).
5. Produce a fully-typed `ResolvedConfig` object: instantiated sources per env,
   one store, notifiers, dashboard packs, slos, mcp settings.

### v0.1 example config

The example config lives at `examples/demo-pack/panoptes.yaml` and selects `dev`
only: `cloudwatch` + `sentry` + `http-health` sources, `victoriametrics` store, a
`logging` notifier, and a two-tier `dashboards` block:

```yaml
  dashboards:
    provider: grafana
    env_variable: true
    core_packs: [errors-sentry, logs, overview]   # tier 1, from core/dashboards/
    consumer_pack:
      path: ${CONSUMER_PACK_DIR}                   # mounted at /packs/consumer (the demo pack)
  mcp:
    transport: stdio                               # v0.1 is stdio; HTTP+SSO is v0.2
    # v0.1-implemented tools only — discovery + query. The v0.2 tools
    # (compare_envs, get_slo, get_cost) are intentionally OMITTED here so the
    # default config path exercises only real tools; if listed, v0.1 registers
    # them as call-time "not available in v0.1" stubs (see § MCP server).
    tools: [describe_signal_catalog, list_dashboards, get_dashboard_data,
            query_metric, search_incidents, search_logs, describe_health]
```

plus the stdio MCP block. `stage`/`prod` are `enabled: false` stubs. The
`consumer_pack.git` variant (repo+ref+subdir) is parsed-but-deferred to v0.2.

> **ADAPTERS §3 vs this v0.1 config — intentional divergence.** The ADAPTERS §3
> config sketch shows the **full (v0.2-superset) `mcp` block** (`transport: http`,
> `auth: sso`, and a `tools:` list including `get_slo`/`compare_envs`). This v0.1
> example uses `transport: stdio` and lists only the v0.1-implemented tools. Both
> are correct for their version; ADAPTERS is the founding full surface, this is the
> v0.1 build set.

## Authorization Rules

v0.1 is local-only; full SSO/OIDC gating is v0.2. The v0.1 contract:

- **Read into observed systems (IAM.md — the single credential domain):** the
  collector uses the local dev AWS profile (or an `assume_role_arn` if provided)
  holding **read-only** CloudWatch + Logs permissions, and a Sentry **read** API
  token. No write actions to any observed system anywhere in the codebase.
  **The authoritative control is the read-only IAM role** (IAM.md hard rule:
  zero `Put*`/`Create*`/`Update*`/`Delete*`/`Write*` on any observed resource);
  the grep guard below is **defense-in-depth**, not the primary gate, and is
  necessarily heuristic. v0.1 has no IAM policy yet, so the guard applies the same
  intent at the client call site.
  **CI guard (`test_no_write_actions_guard`):** greps the core + consumer source
  for the **boto3-shaped mutation calls** that would write to an observed AWS
  system, and is designed to catch the real risky verbs rather than a fixed
  denylist of five prefixes (a tiny denylist silently misses `s3.upload_file`,
  `sns.publish`, `sqs.send_message`, `lambda.invoke`, `ec2.terminate_instances`,
  `dynamodb.batch_write_item` / `transact_write_items`, `iam.attach_role_policy`,
  `cloudwatch.set_alarm_state`). Two complementary checks:
  - **(a) snake_case boto3 mutation shape** — match a genuine-AWS-mutation verb
    set, anchored on a method call shape that **requires a trailing `(`** so plain
    identifiers, attribute names, and prose do not false-positive. The open paren
    follows the method name on the same token in idiomatic boto3, so this still
    catches kwargs/multi-line calls. To catch mid-name forms, anchor on the call
    shape `\.[a-z_]*(put_|create_|update_|delete_|write_|upload_|copy_|publish|send_message|send_raw_email|invoke|terminate_|reboot_|attach_|detach_|enable_|disable_|untag_|associate_|disassociate_|deregister_|batch_write|transact_write|modify_|set_alarm|start_instances|stop_instances|stop_db_instance|put_object|put_item)[a-z_]*\(`.
    This catches real boto3 writes (`s3.put_object()`, `ddb.put_item()`,
    `dynamodb.batch_write_item(...)`), since boto3 Python methods are snake_case.
    **Two distinct non-flagged categories — DO NOT conflate them (this drives the
    self-test buckets below):**
    - **Matched-then-suppressed (the only members):** Python file-I/O `write_text`
      / `write_bytes`. These genuinely contain the matched `write_` verb substring
      and **would** match the regex, so they MUST be added to an explicit
      suppression allowlist (needed because the Phase-5 Grafana provider writes
      JSON to local disk, Risk R6).
    - **Never-matched (no allowlist entry needed):** the read-only CloudWatch Logs
      Insights actions `start_query` / `stop_query` / `get_query_results`, and the
      core/pack registration helpers `register_tools` / `register` / `add_tool` /
      `add_source`. These are read-only / Panoptes-registration methods that must
      never be flagged — but the verb set above **already** omits the bare
      `register_`/`add_`/`start_`/`stop_`/`set_`/`run_`/`tag_`/`remove_` families
      precisely so these tokens are NEVER matched in the first place. They are in
      the same structural bucket as `writelines` / `writer`: the regex has no verb
      alternative that matches them, so no suppression allowlist entry is required.
      (`register_tools` / `start_query` etc. are still listed in the implementation
      allowlist as documentation of intent, but the allowlist is a no-op for them —
      the verb set never produces a hit to suppress.) IAM.md §A explicitly grants
      `logs:StartQuery`/`logs:StopQuery`/`logs:GetQueryResults` as **read-only**
      actions, so a future Logs-Insights cloudwatch adapter calling
      `client.stop_query(...)` is never a false positive; and `register_tools` is
      **mandated** by the spec (`core/mcp/server.py` defines and calls
      `register_tools(mcp_server)`; the demo pack's `pack.py` defines
      `register_tools`).
  - **(b) PascalCase API-action shape** `\b(Put|Create|Update|Delete|Write)[A-Z]`
    as an optional secondary catch, with a documented allowlist for known-safe
    substrings (`UpdatedAt`, `CreatedAt`, `DeleteMarker`, `Writer`) so
    read-response field names / type names do not false-positive. (boto3 Python
    call sites are snake_case, so this shape mainly guards docstrings/comments; it
    may be dropped entirely in favor of (a) — `fb` picks one and documents the
    choice.)
  - **Coverage scope (honest framing) + known misses:** the guard targets
    **boto3-shaped** mutations only. It does **not** inspect raw `httpx`
    `POST`/`PUT`/`DELETE` bodies, so it cannot catch a write performed via a direct
    HTTP call to an upstream mutation endpoint — the `sentry` and `http-health`
    sources use `httpx`, not boto3. **Documented known-miss shapes** (so the
    heuristic's blind spots are explicit, not silent): (i) any mutation issued via
    a **paginator or `.get_paginator(...).paginate(...)`** wrapper rather than a
    direct verb call; (ii) a mutation invoked **dynamically** via
    `getattr(client, verb)(...)` or `client.meta.events`-driven calls, where the
    verb is not a literal token; (iii) the low-level **`client.api_call(...)` /
    botocore `make_api_call`** escape hatch; (iv) any new AWS mutation verb whose
    snake_case form is **not** in the verb set (the set is curated, not exhaustive —
    a brand-new service action would need adding). For all of these the read-only
    contract is enforced by the **Source-never-writes principle** plus the
    **read-only IAM role** (IAM.md hard rule — the authoritative control) plus the
    **upstream API token being read/viewer-scoped** (the least-privilege token
    discipline in the plan's `.env.example`), not by this grep.
  Mirrors IAM.md "a CI check should fail if any non-read action appears." The grep
  is **heuristic defense-in-depth**; the authoritative control is the read-only
  IAM role.
- **MCP (stdio, local) — fully read-only:** no network auth in v0.1 (stdio is
  local-process). The SSO/OIDC proxy is explicitly v0.2. No anonymous *HTTP*
  surface is exposed in v0.1. There is **no write path of any kind** — no
  dashboard authoring, no save-to-repo, no `PanoptesArtifactWriter` credential.
- **No repo-write credential.** New dashboards are authored as code in the
  consumer repo and injected at deploy time; the only deploy-time repo access is
  a **read-only `git fetch`** of the consumer pack at a pinned ref (v0.2 hosted
  path), which is deployment plumbing, not a runtime write.

## Performance Constraints

- **Zero cloud cost** (hard constraint): VictoriaMetrics single-node, Grafana
  OSS, collector, MCP — all local containers. No managed services in v0.1.
- Collector poll cadence default 60s (Open Question 1); `--once` mode for CI so
  integration tests don't loop.
- **Per-source fetch timeout:** each `fetch` is bounded by `fetch_timeout_seconds`
  (default 30s, overridable per source in `config`) so a hung upstream never stalls
  the cycle; on timeout the source is abandoned for that cycle and the loop
  continues.
- **Chronically-down throttle:** the collector keeps a per-source consecutive
  failure counter; after `K` consecutive failures (default `K=3`) error logging for
  that source drops to once per `N` cycles (default `N=10`) so a persistently down
  upstream does not emit one error every poll. The counter resets on a successful
  fetch. The test seam drives `M` cycles via `--once` in a loop (or a monkeypatched
  clock) and asserts a bounded log-call count — never a wall-clock sleep.
- CloudWatch `GetMetricData` and Logs `FilterLogEvents` are paginated; the
  cloudwatch source must page fully and bound the window (default last 15m per
  poll) to stay within free-tier-friendly API call volume.
- Sentry REST calls are rate-limited; the sentry source must respect
  `Retry-After` and **surface the HTTP response body on failure** (via a
  `_format_failure` helper): an opaque 4xx with only URL + status code otherwise
  costs a full debug cycle to learn which field/value the upstream rejected. This
  is Panoptes' own REST-client convention.
- VM queries from MCP tools and Grafana hit the same endpoint; no special
  performance tuning required at v0.1 data volumes.

## CI/CD

Panoptes uses a conventional layered GitHub Actions shape — a **composite setup
action**, a **reusable check workflow**, an **entry workflow** that is
change-aware via `dorny/paths-filter` (lint/type/unit always run; the integration
job is conditionally skipped for docs-only changes) plus `concurrency`, and a
final **merge-gatekeeper** — scoped to a single-language Python OSS repo. All
gates run on PR and on push to `main`. This CI is self-contained in the Panoptes
repo (see the "Self-contained" note below).

### Composite action — `.github/actions/setup-panoptes/action.yml`
A thin per-language setup action: pin Python, cache pip, install dev deps once.

```yaml
name: 'Setup Panoptes Toolchain'
description: 'Python + cached dev dependencies'
inputs:
  python-version: { description: 'Python version', default: '3.12' }
runs:
  using: 'composite'
  steps:
    - uses: actions/setup-python@v6
      with: { python-version: '${{ inputs.python-version }}', cache: 'pip' }
    - shell: bash
      run: |
        python -m pip install --upgrade pip
        pip install -e '.[dev]'   # ruff, mypy, pytest, pytest-cov, testcontainers
```

### Reusable checks — `.github/workflows/ci-checks.yml`
`workflow_call`; one job per gate so failures are isolated.

```yaml
name: '🛠️ CI Checks'
on:
  workflow_call:
    inputs:
      run-integration: { type: boolean, default: false }
permissions: { contents: read }
jobs:
  lint-type:                       # ruff lint + format-check + mypy --strict
    name: 🐍 Lint & Types
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: ./.github/actions/setup-panoptes
      - run: ruff check .
      - run: ruff format --check .
      - run: mypy --strict .

  unit:                            # fast; the coverage gate
    name: 🧪 Unit
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: ./.github/actions/setup-panoptes
      # Global floor measures core only. examples/ is a test fixture (the demo
      # pack), not shipped core — excluded from the global gate so a fixture
      # coverage dip can neither mask nor inflate the core number.
      - run: pytest -m "not integration" --cov=core --cov-fail-under=85
      # Authoritative per-package floor (spec § Tests Coverage mandate): the two
      # areas the design docs single out. This INTENTIONALLY re-executes the full
      # unit suite a second time to recompute coverage scoped to core.sources +
      # core.mcp; --cov-append keeps the second run from re-clobbering the first
      # run's .coverage data file. The scoped percentage is therefore measured over
      # the union of both runs' data (i.e. all unit tests touching those packages),
      # not only sources/mcp-specific tests — this is expected. Trade-off accepted:
      # the second full run roughly doubles unit-job wall time AND doubles the flake
      # exposure of any nondeterministic unit test (so unit tests MUST be strictly
      # deterministic — they are, given respx/botocore.stub everywhere) but keeps a
      # single obvious gate command. Do NOT "optimize" it into a `coverage report
      # --include=` post-step without re-verifying both floors still gate
      # (`coverage combine` + two scoped `coverage report --include=` is the
      # documented v0.2 optimization; the spec warns against doing it in v0.1).
      - run: pytest -m "not integration" --cov=core.sources --cov=core.mcp --cov-append --cov-fail-under=80

  guards:                          # the standalone-purity invariants
    name: 🛡️ Boundary Guards
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: ./.github/actions/setup-panoptes
      - run: pytest tests/unit/test_core_purity_guard.py tests/unit/test_no_write_actions_guard.py

  integration:                     # compose/testcontainers; marker-gated, opt-in
    name: 🐳 Integration (compose)
    if: ${{ inputs.run-integration }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: ./.github/actions/setup-panoptes
      # Phases 0-7 collect zero integration tests; pytest exit code 5 ("no tests
      # collected") would otherwise red-bar the skeleton. Tolerate exit-5 until
      # Phase 8 lands real integration tests — then this tolerance is REMOVED
      # (see § Rollout Phases Phase 8 exit criteria) so an empty integration
      # collection becomes a failure again and the gate cannot rot to always-green.
      - run: pytest -m integration || [ $? -eq 5 ]   # testcontainers spins VM+Grafana; Docker preinstalled on ubuntu-latest
```

### Entry — `.github/workflows/ci.yml`

```yaml
name: 🔍 CI
on:
  pull_request: { branches: [main] }
  push: { branches: [main] }
  workflow_dispatch:
concurrency:
  group: ci-${{ github.event.pull_request.number || github.ref }}
  cancel-in-progress: true
permissions: { contents: read }
jobs:
  changes:                         # dorny/paths-filter: skip integration for docs-only
    runs-on: ubuntu-latest
    outputs: { code: ${{ steps.f.outputs.code }} }
    steps:
      - uses: actions/checkout@v6
      - uses: dorny/paths-filter@v3
        id: f
        with:
          filters: |
            code:
              - 'core/**'
              - 'examples/**'
              - 'tests/**'
              - 'docker-compose.yml'
              - 'pyproject.toml'
  ci:
    needs: changes
    uses: ./.github/workflows/ci-checks.yml
    with:
      run-integration: ${{ needs.changes.outputs.code == 'true' }}
  merge-gatekeeper:
    name: 🚪 Merge Gatekeeper
    needs: [ci]
    if: ${{ always() }}              # run even when ci failed, so this is a real red check, not a skip
    runs-on: ubuntu-latest
    steps:
      - run: |
          if ${{ contains(needs.*.result, 'failure') }}; then
            echo "A required check failed."; exit 1
          fi
          echo "All checks passed."
```

> **Branch-protection wiring (intended):** mark **Merge Gatekeeper** as the single
> required check. It uses `if: always()` and exits non-zero on any upstream
> failure, so a failing `ci` produces a genuinely red gatekeeper rather than a
> neutral/skipped status that branch protection would treat as satisfied. (If you
> prefer, require the underlying `lint-type`/`unit`/`guards`/`integration` jobs
> directly instead.)

**Notes:** single language → no node/flutter jobs; the change-awareness is via
`dorny/paths-filter` (not `on: paths-ignore`), so on a docs-only change the
`lint-type`/`unit`/`guards` jobs still run and only the integration job is
conditionally skipped; the two boundary guards (`test_core_purity_guard`,
`test_no_write_actions_guard`) get their own visible job so a coupling regression
is an obvious red check, not buried in the unit run.

> **`code` filter coupling (intentional — keep through edits):** the `changes`
> job's `code` filter deliberately includes `tests/**` (alongside `core/**`,
> `examples/**`, `docker-compose.yml`, `pyproject.toml`). This is load-bearing:
> a tests-only PR that would break integration still flips `code == 'true'`, so
> `run-integration` stays `true` and the integration job runs against it. If a
> future edit drops `tests/**` from the filter, a test-only change could skip the
> required integration check (a skipped required job counts as non-failure to the
> merge-gatekeeper's `contains(needs.*.result, 'failure')`), letting an
> integration-breaking test merge unverified. Keep `tests/**` in the filter.

**Self-contained.** This CI lives entirely in the Panoptes repo and is **not
connected to any consumer's CI** — no shared workflows, no `check_all.py`, no
reusable-workflow calls into another repo. It borrows structure and concepts
from prior art but shares no files. A consumer's pipeline and Panoptes' pipeline
run fully independently.

## Local run & IaC contract (locked)

- **Local = docker-compose.** `docker-compose up` brings up the full stack

  reserves `deploy/terraform/` so the module slots in without restructuring.
  See § Out of Scope.

    leaks into a test body and the no-async-in-tests rule holds. The
    testcontainer VM URL is passed to the server via env. `fb` uses this
    mechanism rather than improvising async into the suite.
  - **Flake prevention (provisioning is async):** the Grafana test must **poll**
    `GET /api/search` with bounded retry/backoff until all four expected dashboard
    UIDs/titles appear — never a single immediate assertion after container start.
    Likewise `test_collector_to_vm` polls VM readiness (`/health` or
    `/api/v1/query` → 200) before the PromQL read-back.
  - **Parity assertion (the literal proof of "two faces, one store"):** write a
    **single synthetic sample at a fixed timestamp** for a known `panoptes_*`
    series, then run BOTH a direct PromQL `/query_range` and the panel's PromQL
    through `get_dashboard_data(...)` over stdio. **Pin both callers to the SAME
    endpoint family** (`/query_range` with identical `start`/`end`/`step` — never
    mix `/query_range` against an instant `/query`) and the **identical
    `TimeWindow` + `step_seconds`** on both sides. **The PRIMARY assertion is the
    scalar last-value** — assert the value at the sample's own grid point is exactly
    equal across both callers. This is the robust path: VictoriaMetrics
    staleness/last-value carry-forward can repeat a single sample across multiple
    interior `/query_range` grid points, so a full rounded-tuple-SET comparison can
    diverge even when the data is identical. The `(rounded-timestamp, value)`
    tuple-set comparison (trailing empty/stale sample dropped) is an **optional
    secondary** check, compared at the sample's own grid point — not the full grid.
    Rounding (not raw float-formatted series) keeps it deterministic when used.
  - **Negative end-to-end:** ask the live MCP for traces over stdio and confirm
    the explicit "no trace source" error surfaces over the real transport (not
    just in the unit fake).
- **Phase 8 exit criterion — re-tighten the integration gate (one concrete
  mechanism, not a choice):** now that real integration tests exist, **both**
  (a) **remove** the `|| [ $? -eq 5 ]` exit-code-5 tolerance from the CI
  integration job **and** (b) add an explicit minimum-collected-count assertion —
  `pytest -m integration` must collect **≥ 5 tests**: `test_collector_to_vm`,
  `test_grafana_provisioning`, and the three distinct assertions of
  `test_mcp_stdio_e2e` written as **three separate test functions**
  (discovery+health e2e, Grafana↔MCP parity, negative no-trace-source). The floor
  is keyed to the **assertion count, not the file count**, so a future
  consolidation of two integration files cannot quietly shrink coverage below the
  floor while still passing a file-keyed `≥3` check. Doing both closes the anti-rot
  gap unambiguously: removing only the exit-5 tolerance would still let a collection
  error under a non-5 exit path silently reduce the suite, and asserting only the
  count without removing the tolerance would still mask exit-5. The count floor is
  the authoritative anti-rot guard; the tolerance removal restores the exit-5 red
  bar.

## Tests (the contract — none deferred)

### Unit (mocked upstreams; fast; default CI job)
- **Model:** every signal type constructs — **including `TraceSignal`** — and the
  mandatory `env` validator fires on **each** type (a `TraceSignal` built without
  `env` raises in `__post_init__`, same as the others); union discrimination by
  `kind`; enum round-trips.
- **Registry:** register + build; unknown type fails fast and lists available
  types; mypy-level plane conformance (a non-conforming class is a type error).
- **Config:** `${VAR}` interpolation; missing var fails fast naming the var;
  unknown adapter type fails fast; `enabled: false` env inert; full
  `ResolvedConfig` shape for the reference example. **`provides:` ↔
  `capabilities()` reconciliation:** a source config whose declared `provides:`
  list **disagrees** with the built adapter's `capabilities()` (e.g.
  `provides: [incident]` for a `sentry` source whose `capabilities()` returns
  `{incident, metric}`) **fails fast at config-resolve time** naming the source +
  the mismatch — `capabilities()` is authoritative, `provides:` is advisory
  documentation that must agree (ADAPTERS §3 Conventions). This forecloses the
  silent capability-mis-routing class. **Negative paths:** malformed
  / invalid YAML raises a clear `PanoptesError` (parse error names the problem); a
  config with **all** environments `enabled: false` resolves to an **empty**
  live-adapter set and the collector run is a clean no-op (no divide-by-zero, no
  crash); a `ConsumerPackConfig` with **neither** `path` **nor** `git` (or both at
  once) is rejected by the union validation; a v0.1 config supplying **only `git:`**
  **parses and validates successfully** (the shape is valid — `git` is
  parsed-but-deferred to v0.2, not rejected at parse time), but when `git` is the
  active consumer-pack selector the **Grafana provider raises a clear
  `CapabilityError("git injection is v0.2; use `path` in v0.1")`** at provision
  time — so the deferred-but-parsed boundary is explicit and testable (parse OK,
  act-on-it fails in v0.1); an unknown `core_packs` id that does not resolve under
  `core/dashboards/` fails fast naming the missing pack.
- **Stores:** VM write line-format + PromQL parse → `MetricSeries` (httpx
  mocked); passthrough no-op + query `CapabilityError`; `env` label fidelity.
  **Negative paths:** a VM write/query against a 5xx / connection-error **surfaces
  the response body via `_format_failure` and raises** (never silently swallowed)
  — exercise **both** `_format_failure` branches with `respx`: the 5xx-with-body
  case via `.respond(500, text=body)` and the connection-error case via
  `.mock(side_effect=httpx.ConnectError(...))`, not only the HTTP-status path; a
  query against an empty/200-no-data response returns an **empty
  `list[MetricSeries]`**, not an error.
- **Sources:** per adapter, recorded upstream payload → exact normalized signals
  + derived metrics + capability set; cloudwatch pagination; sentry `Retry-After`
  + failure-body surfaced; http-health up/down/latency. boto3 via `botocore.stub`.
  **Derived-metric labels (exact):** assert each derived metric carries its
  **exact** label set — `panoptes_sentry_incident_count` → `env`+`level`+`project`;
  `panoptes_log_error_rate` → `env`+`log_group`; `panoptes_health_up` /
  `panoptes_health_latency_ms` → `env`+`url` (a missing label is a silent
  dashboard-empty regression). **Negative paths:** sentry non-429 4xx/5xx (401 bad
  token, 404 wrong project) → a typed error, **not** an empty incident list;
  http-health connection-refused vs timeout vs 5xx all map to `_up=0` with latency
  recorded; cloudwatch STS `AssumeRole` denial (expired/denied) surfaces a clear
  auth error and does **not** crash the whole collector cycle.
  **TRACE-absence invariant (locks the no-trace-source precondition):** assert the
  **union** of `capabilities()` across all v0.1-registered sources is **exactly**
  `{METRIC, LOG, INCIDENT}` — TRACE is provided by **no** v0.1 source. This is the
  precondition that makes the "no trace source" negative path (MCP + end-to-end)
  true; a single targeted guard catches a future adapter accidentally adding TRACE
  to a capability set, rather than letting the no-trace tests start failing
  confusingly.
  **AssumeRole boundary + mock fidelity (so the does-not-abort claim is actually
  exercised, not just asserted):** the cloudwatch adapter attempts assume-role
  inside **`health()`** (credential resolution is a reachability concern), so an
  AssumeRole denial manifests through the **same `health()` try-continue
  boundary** the collector test exercises — that is what ties this to the Phase-4
  "`health()` failure does not abort the run" rule. Because AssumeRole runs on a
  **separate `sts` client** (not the cloudwatch client), the adapter MUST expose an
  **injectable seam** so the test can target the exact client the stub attaches to:
  both the `sts` client and the cloudwatch client are obtained via an overridable
  factory/constructor parameter (e.g. an optional `sts_client` / a `_assume_role()`
  method `fb` can monkeypatch) — they are NOT constructed inline inside
  `fetch()`/`health()` via a bare `boto3.client("sts")` with no seam (which would
  make `botocore.stub.Stubber` unable to attach, leaving the test unwritable). With
  the seam, the unit test stubs the sts client with
  `Stubber.add_client_error("assume_role", "AccessDenied")` (or monkeypatches
  `_assume_role()`); stubbing the cloudwatch client alone never reaches the
  assume-role branch. The collector negative-path test injects this
  AssumeRole-denial fake at the `health()` boundary so the "does not crash the
  whole cycle" assertion runs end-to-end.
- **Collector:** per-env per-source dispatch; disabled env skipped; one source
  failing `health()` does not abort the run; signals reach the store. **Negative
  paths:** a `store.write()` raising (VM unreachable) is **caught per-batch,
  logged, and the loop continues** — the next env is still processed; a per-source
  fetch has a **bounded timeout** — `fetch_timeout_seconds` (default 30s,
  config-overridable per source); a test patches a source whose `fetch` exceeds it
  and asserts the fetch is abandoned and the loop continues to the next source/env;
  a chronically down upstream is **log-throttled** via a per-source consecutive
  failure counter — after `K` consecutive `health()`/`fetch` failures (default
  `K=3`) error logging drops to **once per `N` cycles** (default `N=10`); a test
  monkeypatches a perpetually-failing source, drives `M` cycles (via `--once` in a
  loop or a monkeypatched clock — never a wall-clock sleep), and asserts the
  error-log call count over `M` cycles is bounded (not one-per-poll).
- **Notifier:** `logging` notifier renders the alert (f-string, no `%s`); a config
  with an **unregistered** notifier `type` (e.g. `pagerduty`) **fails fast** with
  `UnknownAdapterError` listing the available notifier types — closing Open
  Question 6's "unknown notifier types still fail-fast" claim on the notifier
  plane specifically.
- **Dashboards:** every shipped core JSON is valid, has the `env` template var,
  references only known metrics; **a dashboard-validity assertion that each core
  panel's PromQL references only labels the corresponding derived metric actually
  emits** (guards against a label drift that empties a panel); **injection** — a
  temp consumer dir's `dashboards/demo/dashboard.json` is globbed + provisioned
  alongside core packs.
- **MCP:** each tool aggregates correctly over fake store/sources; capability
  negotiation returns explicit "no source for X" (the canonical case uses **TRACE**
  as the "no source configured" kind); `env="all"` fan-out; precise `TypedDict`
  returns; explicit failure (never silent-empty). **`describe_health` surfaces an
  unreachable source (the tool's core promise):** `describe_health(env)` with a
  source whose `health()` returns `reachable: false` **includes that source in the
  rollup marked unreachable** — it is neither omitted nor returned as a
  silent-empty rollup, so "the one thing to look at" actually shows the thing that
  is down. **Negative / edge paths:**
  `get_dashboard_data(id=…)` for an **unknown** dashboard id → an explicit
  `CapabilityError` (the spec defines no separate `NotFoundError`; an unknown
  dashboard id raises `CapabilityError(detail=…)` — never silent/None); the
  **`query_metric` store-reading tool** against a configured **`passthrough`**
  store surfaces the store's `CapabilityError` as a **structured MCP error** —
  never a crash or silent-empty — so a source-only (passthrough) misconfiguration
  fails explicitly (this is the plausible "operator ran source-only" case; the test
  builds a `ResolvedConfig` with `store.type: passthrough` directly — `passthrough`
  is the tests/source-only store and is never the example config's active store,
  so the fixture is synthetic, mirroring the R14 inline-fixture note);
  `env="all"` where one env's
  required source is down/unconfigured → a **per-env partial result with an
  explicit per-env error marker** (the call does not wholesale-fail), tested; a
  tool listed in config but **not implemented in v0.1** (`compare_envs`, `get_slo`)
  is registered as a **stub** that returns an explicit "not available in v0.1
  (ships v0.2)" MCP error **at call time** (NOT a config-resolve failure — these
  are valid future tools, not unknown adapter types; see § MCP server contract),
  asserted by listing a v0.2 tool and invoking it. **Read-only contract (structural, not name-denylist):**
  with the **default v0.1 config** (no v0.2 stubs listed), assert the registered
  tool set is **exactly** the known read-only set (catalog / query / discovery
  tools + the injected pack's declared tools) **and** that no registered tool name
  matches a mutation-verb regex
  (`create_`/`save_`/`update_`/`delete_`/`set_`/`post_`/`ack_`/`silence_`) — so a
  future write tool under any name fails the test, not only the two hardcoded
  `create_dashboard`/`save_dashboard` strings. (When a config DOES list a v0.2
  tool, its registered stub is read-only-by-error and likewise does not match the
  mutation-verb regex — `compare_envs`/`get_slo`/`get_cost` are read-shaped names.)
- **Demo pack:** loads via the injection hook without core importing it;
  `get_demo_signal` produces `DemoSignal`; demo dashboard valid. The
  `get_dashboard_data` test uses a small **inline/`tmp_path` dashboard JSON
  fixture** with known `panel.targets[].expr` — it does **not** depend on the
  Phase-5 shipped `core/dashboards` JSON, keeping the Phase-6 unit test
  self-contained.
- **Guards:**
  - **Core-purity (structural + brand-free):** no `core/**/*.py` contains
    `from examples` / `import examples`; defense-in-depth grep finds zero generic
    consumer-domain identifiers (`allocator`, `matchmaking`, `agones`) in `core/`.
    The guard embeds **no brand literal**.
  - **Additive-injection runtime invariant:** after loading the demo pack via the
    `PANOPTES_CONSUMER_PACK` hook, assert (a) no `core.*` module now references an
    `examples/` symbol; (b) the core registries contain the consumer-registered
    adapter/tool **only as an addition**, with core's own registrations unchanged;
    (c) re-running with `PANOPTES_CONSUMER_PACK` unset yields a core-only registry
    **identical** to the no-pack baseline (injection is purely additive and
    reversible).
  - **No-write-actions (self-testing):** in addition to scanning the codebase,
    feed the matcher synthetic known-bad strings (`ddb.put_item(`,
    `client.create_table(`, `s3.upload_file(`, `sns.publish(`,
    `dynamodb.batch_write_item(`, `ec2.terminate_instances(`) and assert it flags
    **all** of them. (If the optional PascalCase shape (b) is retained, an
    `s3.PutObject`-shape is also asserted flagged; if `fb` drops the PascalCase
    shape in favor of the snake_case set, drop that assertion and use
    `s3.put_object(` instead — see the verb set, which covers `put_object`.) Then
    feed it the non-flagged forms and assert each is **not flagged** for the right
    reason. **Critical — match these buckets to regex reality (the verb set
    deliberately omits the bare `register_`/`add_`/`start_`/`stop_` families):**
    - **matched-then-suppressed** (the ONLY members — these DO contain a matched
      `write_` verb substring and would match without the allowlist):
      `path.write_text(`, `path.write_bytes(` — file-I/O that MUST NOT be flagged,
      so they require an explicit suppression allowlist entry;
    - **never-matched** (the verb set produces NO hit for these — assert
      `pattern.search(...) is None`, exactly as for `writelines`/`writer`):
      `client.stop_query(`, `client.start_query(`, `client.get_query_results(`,
      `mcp.register_tools(`, `pack.add_tool(`, `provider.add_source(`,
      `f.writelines(`, `csv.writer(`. The verb set has no alternative that matches
      `stop_query`/`start_query`/`get_query_results` (no `stop_`/`start_`/`get_`
      bare family) or `register_tools`/`add_tool`/`add_source` (no bare
      `register_`/`add_` family), so all of these are never matched in the first
      place — proving the matcher's shape, not a vacuous allowlist hit.
    This keeps the guard's own logic regression-safe — both that it flags real
    boto3 mutations AND that the verb set is built so it never collides with
    Panoptes' own `register_tools`/`add_tool` symbols or the read-only
    Logs-Insights actions IAM.md grants (they are never matched, not suppressed).

### Integration (compose / testcontainers; dedicated CI job; marker-gated)
- Collector `--once` → VictoriaMetrics → PromQL read-back of written series (poll

- Grafana datasource reachable + all four dashboards provisioned/loaded — assert

  series, then run a direct PromQL `/query_range` and `get_dashboard_data(...)`
  over stdio — **both pinned to `/query_range`** with the **identical `TimeWindow` +
  `step_seconds`** (never one `/query_range` vs an instant `/query`). **The primary
  assertion is the scalar last-value at the sample's own grid point** being exactly
  equal across both callers (robust against VM staleness/last-value carry-forward,
  which can repeat the sample across interior grid points). An optional secondary
  assertion compares the normalized `(rounded-timestamp, value)` tuple at that same
  grid point — not the full grid. This makes the test deterministic, not a racy
  raw-series or full-grid equality.
- **Negative end-to-end:** ask the live MCP for traces over stdio → the explicit
  "no trace source" error surfaces over the real transport.
- Integration tests use **synthetic `panoptes_*` series only** (hermetic CI per
  Risk R10) — no live AWS/Sentry.
- **Anti-rot floor:** from Phase 8 the integration job both drops the exit-5
  tolerance and asserts `pytest -m integration` collects **≥ 5 tests** — keyed to
  the assertion count, not the file count: `test_collector_to_vm`,
  `test_grafana_provisioning`, and the three **separate test functions** of
  `test_mcp_stdio_e2e` (discovery+health e2e, parity, negative no-trace-source).
  Tying the floor to assertion granularity means a future merge of two integration
  files cannot quietly shrink coverage under the floor; a collection error can
  never silently green the gate (see Phase-8 exit criterion).

### Coverage mandate
- **Per-package floor (authoritative):** ≥80% line coverage on `core/sources/*`
  (normalization) and `core/mcp/*` (aggregation) — the two areas the design docs
  single out — enforced by a dedicated `--cov=core.sources --cov=core.mcp
  --cov-fail-under=80` run. **Measurement semantics (honest framing):** because the
  CI unit job runs this with `--cov-append` onto the prior full-suite run, the
  per-package floor is measured over **all** unit tests that exercise those
  packages, not only their dedicated test files — incidental coverage (e.g. a
  collector or config test that imports a source) **counts** toward the floor. This
  is the accepted v0.1 trade-off (one obvious gate command over a perfectly-scoped
  measurement); the dedicated `test_source_*` / `test_mcp_*` suites are still the
  substance of that coverage. A scoped `coverage run -m pytest tests/unit/test_source_*.py
  tests/unit/test_mcp_*.py` measurement is a documented v0.2 tightening, not v0.1.
- **Global floor:** ≥85% line coverage on **`core` only** — enforced by
  `--cov=core --cov-fail-under=85`. **`examples/` is excluded from the global
  floor**: it is a test fixture (the demo pack), not shipped core, so a fixture
  coverage dip can neither mask nor inflate the core gate. Both numbers live in
  the `## CI/CD` unit job (the spec is the single source of truth for the gate
  numbers).

## Out of Scope (v0.1 — deferred to v0.2/v0.3)

- **Terraform module** + always-on hosted home (v0.2). Repo reserves
  `deploy/terraform/` so the module slots in cleanly without restructuring. The
  module runs the *same* `docker-compose.yml` v0.1 ships, so local == deployed.
- **MCP over streamable HTTP + SSO/OIDC gating / auth proxy** (v0.2). v0.1 is
  stdio-only, local-process.
- **MCP dashboard-authoring write path — DROPPED entirely** (not deferred).
  There is no `create_dashboard` / `save_dashboard`, no Git-PR round-trip, and no
  artifact-write credential in any version. New dashboards are authored as code
  in the consumer's own repo and injected at deploy time; the MCP server is fully
  read-only. Panoptes only *provisions* dashboard JSON (core + injected consumer).
- **`kubernetes`, `prometheus`, `loki`, `tempo/jaeger`, `datadog` core sources**
  (v0.2/v0.3). A **game-server fleet source** (e.g. Agones) is a **consumer-pack
  source adapter**, not a core source — it is domain-specific and lives in the
  consumer's repo (it backs the consumer-tier Fleet dashboard C2). The Fleet (C2)
  and Game/Business (C3) **consumer** dashboards depend on these and are deferred /
  consumer-owned. (Keeping `agones` out of core is also why the core-purity guard
  bans it as a generic consumer-domain token — see § Strategic positioning.)
- **`sns` / `slack` notifiers** + declarative alert rules + SLO evaluation
  (`get_slo`, `compare_envs`, `get_cost`, `get_fleet_health`, `get_game_metrics`).
  v0.1 ships only the `logging` notifier; the Notifier plane is proven.
- **Root-cause / causal-chain tracing** — explicitly DROPPED from the product;
  do not plan it.
- **`stage` / `prod` live wiring** — present only as `enabled: false` config stubs.
