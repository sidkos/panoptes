## The hosting model — dedicated EKS in the SAME account, and exactly which failure domains are independent

The single most important framing in v0.2:

> **Panoptes' hosted home is its OWN dedicated EKS cluster + its OWN dedicated VPC,
> running in the SAME AWS account as the observed infra, and it is NEVER deployed inside
> any observed cluster.** Failure-domain independence (ARCHITECTURE §1, the load-bearing
> rule) is preserved at the **cluster and network layers** — a separate, dedicated
> Panoptes cluster in a separate VPC, never the observed one — so a cluster crash,
> node-group failure, or VPC-network fault in an observed environment cannot take
> Panoptes down with it. What is **NOT** independent is the **AWS account control
> plane**: Panoptes shares the account with the observed infra (a deliberate trade — see
> the Decision below), so an account-level event (root-credential compromise,
> account-wide service-quota exhaustion, an account suspension) is a shared blast radius.

> **`Decision: run Panoptes in the SAME AWS account as the observed infra (dedicated
> VPC + dedicated cluster, NOT a separate account) because the operator needs the
> existing AWS-profile / assume-role multi-env access mechanism (the v0.1 AWS_PROFILE +
> PANOPTES_ASSUME_ROLE_ARN config seam) to keep working without cross-account trust
> plumbing; we keep a dedicated VPC + dedicated cluster so cluster-level and
> network-level failure-domain independence are preserved (Panoptes never shares a
> cluster or VPC with what it watches), and we accept the shared-account control-plane
> blast radius as a deliberate, documented trade. because the failure-domain rule's
> sharpest edge — "don't die with the thing you monitor" — is satisfied by cluster+VPC
> isolation; full account isolation is the stronger form but would break the
> profile-based access the operator requires, and is a clean future hardening (move
> Panoptes to its own account + cross-account assume-role) if the shared-account risk
> ever outweighs the access convenience.`**

**Local vs hosted (the deploy split — keep both):**

| | Local (v0.1, unchanged) | Hosted (v0.2, NEW) |
|---|---|---|
| Shape | `docker-compose up` on a laptop | dedicated EKS cluster, K8s workloads (Helm) |
| MCP transport | stdio | streamable HTTP behind a GitHub-gated ingress |
| Cross-env read auth | local AWS profile / assume-role | **IRSA** base identity → AWS-profile / assume-role per env (SAME account) |
| TLS / SSO | none (local process) | nginx ingress + cert-manager (Let's Encrypt) + oauth2-proxy (GitHub) |
| Cost | zero cloud cost | minimal right-sized cluster (honest step-up) |
| Image | same GHCR image | same GHCR image |

`Decision: keep docker-compose as the local path and add EKS as the hosted path (rather
than replacing compose) because the local zero-cost proof loop is load-bearing for
development + the v0.1 integration suite, and the SAME GHCR image runs in both — so
"works locally" still means "works hosted" at the container level. The two differ only
in orchestration (compose vs K8s) and the edge (stdio vs GitHub-gated nginx ingress).
because the v0.1 non-negotiable "local dev loop is the same stack" survives the EKS move at the image
boundary even though the orchestration diverges.`

**Cost note (honest).** A dedicated EKS cluster is a real cost step-up over the prior
single-`t4g.small`-EC2 + compose idea (EKS control-plane hours + a node group + an nginx
ingress LB). It is accepted because it is the cluster/VPC-failure-domain-independent,
IRSA-native, GitHub-gated home the hosted single pane needs. It is kept minimal: a single
small **managed node group** (spot-eligible, single-AZ acceptable for a dev/home stack), a
single-node store, no Agones, no game-server machinery, no multi-AZ HA, no Karpenter.

## Locked decisions (operator-confirmed 2026-06-03 — FIRM, not defaults)

The six hosting choices are **decided**. They are recorded here as firm
`Decision: … because …` records; `fb` implements them as written (no open forks remain).

1. **Account isolation → SAME AWS account, DEDICATED VPC + DEDICATED cluster; multi-env
   access via AWS profiles + assume-role.** **`Decision: run Panoptes in the SAME AWS
   account as the observed infra, in its OWN dedicated VPC + EKS cluster, because the
   operator needs the existing AWS-profile / assume-role multi-env mechanism (the v0.1
   AWS_PROFILE + PANOPTES_ASSUME_ROLE_ARN config seam) to keep working without
   cross-account trust plumbing — IRSA gives the pod its base identity and it then
   assumes the per-env read-roles WITHIN the same account. We keep a dedicated VPC +
   cluster so cluster-level and network-level failure-domain independence are preserved
   (never the observed cluster/VPC); we accept the SHARED-ACCOUNT control-plane blast
   radius as a deliberate, documented trade. because cluster+VPC isolation satisfies the
   sharpest edge of the failure-domain rule ("don't die with what you monitor"); full
   account isolation is the stronger form but would break profile-based access, and is a
   clean future hardening.`** (See § The hosting model for the full failure-domain
   breakdown — cluster + VPC independent, account shared.)

2. **Nodes → a single small EKS MANAGED node group** (spot-eligible, single-AZ
   acceptable). **`Decision: a managed node group over Karpenter because the stack is a
   fixed, small, always-on set of workloads (store + Grafana + collector + MCP + proxy)
   — there is nothing to autoscale for a one-node monitoring stack; a managed node group
   is the minimal-ops, minimal-cost choice, and Karpenter's controller + provisioner
   complexity buys nothing for a static workload and would undercut cost discipline.`**
   (Karpenter as a Panoptes node provisioner is OUT; the v0.3 `karpenter` core
   *dashboard pack* — which OBSERVES a consumer's or Panoptes' own Karpenter — is a
   separate, unrelated concern.)

3. **Ingress + TLS → nginx ingress + cert-manager (Let's Encrypt) + oauth2-proxy as the
   auth gate.** **`Decision: nginx-ingress + cert-manager + oauth2-proxy over ALB-native
   OIDC because the SSO IdP is GitHub (decision #5), and GitHub is OAuth2 — it does NOT
   expose standard OIDC discovery, so ALB's authenticate-oidc listener action cannot
   gate it. nginx-ingress external-auth (nginx.ingress.kubernetes.io/auth-url +
   auth-signin annotations) forwarding to oauth2-proxy is the canonical GitHub-org-gated
   pattern; cert-manager + Let's Encrypt handles TLS issuance/renewal. because the auth
   mechanism (GitHub OAuth2) dictates the ingress: ALB-native OIDC is unavailable for a
   non-OIDC-discovery IdP, and the nginx forward-auth → oauth2-proxy chain is the
   standard, well-trodden GitHub gate.`** The MCP server code is unaffected — oauth2-proxy
   injects a verified identity header; the server never validates tokens.

4. **Packaging → a Helm chart (`charts/panoptes`).** **`Decision: Helm because the stack
   has real templated values (image tag, hostname, GitHub OAuth client id/secret + org
   allowlist, IRSA role ARN, store retention) that map cleanly to a values.yaml, and
   helm lint + helm template | kubeconform is a clean hermetic CI gate.`** (Kustomize is
   NOT offered — Helm is the confirmed, sole packaging path.)

5. **SSO IdP → GitHub (via oauth2-proxy's `github` provider).** **`Decision: GitHub
   OAuth2 via oauth2-proxy's github provider, restricting access by a GitHub ORG (and/or
   team) allowlist, because the operator's identity already lives in GitHub and an
   org/team allowlist is the natural access boundary for an OSS-flavored tool. This is
   GitHub OAuth2, NOT generic OIDC — oauth2-proxy reads
   GITHUB_CLIENT_ID/GITHUB_CLIENT_SECRET + the --github-org/--github-team allowlist (from
   Helm values / env), and dictates the nginx forward-auth ingress of decision #3
   (ALB-native OIDC cannot gate a non-OIDC-discovery provider).`** The MCP streamable-HTTP
   Service stays **ClusterIP-only**, reachable solely through the oauth2-proxy-gated nginx
   ingress (the anonymous-bypass guard asserted in the Helm-render test). "Two faces, one
   store" reuse of `build_server` is unchanged.

6. **`stage`/`prod` ownership → Panoptes owns only the IRSA role + the parameterized
   assume-role grants; `stage`/`prod` stay `enabled:false` config stubs.** **`Decision:
   Panoptes' Terraform provisions ONLY its IRSA role + a parameterized list of assumable
   per-env read-role ARNs (empty for the disabled envs); it does NOT create the
   read-roles in observed accounts (consumer's IaC, IAM.md §D), because creating roles
   the observed side owns would require write admin Panoptes must not hold — the single
   read-only credential domain. stage/prod remain enabled:false stubs that light up with
   a flag flip + a non-empty role-ARN, no code change.`** (Same-account per decision #1,
   so the assume-role targets are in-account roles, not cross-account.)

## Product Specification

### Overview

v0.2 takes the v0.1 local-compose proof and stands it up in its **own dedicated EKS
cluster**, gated by SSO. It adds:

- A **Terraform module** (`modules/stack/`) that provisions a dedicated EKS cluster
  (control plane + a minimal managed node group) in a **dedicated VPC** (SAME AWS account
  as the observed infra — decision #1), **IRSA** for the collector/MCP service accounts,
  and the **nginx-ingress + cert-manager** prerequisites for the GitHub-gated ingress.
- A **Helm chart** (`charts/panoptes/`) that runs the stack as Kubernetes workloads
  (VictoriaMetrics StatefulSet, Grafana Deployment, collector Deployment, MCP
  Deployment, oauth2-proxy Deployment, Services + an Ingress). The same GHCR image v0.1
  bakes runs the collector/MCP pods.
- The MCP server's **streamable-HTTP face**, exposed through the **GitHub-gated nginx
  ingress** (oauth2-proxy `github` provider, forward-auth). The v0.1 **stdio face stays**
  (local) — HTTP is additive and reuses the SAME `build_server`.
- A core **`kubernetes`** source (consumer-agnostic; also self-observes Panoptes' own
  cluster) + its `get_cluster_state` MCP tool and a core Kubernetes dashboard pack.
- **`sns`** + **`slack`** notifiers and a **declarative alert-rule** model evaluated by
  the collector.
- An **SLO dashboard** + `get_slo` / `compare_envs` (promoting the v0.1 stubs).
- `stage`/`prod` wired as `enabled:false` config stubs.
- A **GHCR container-image publish pipeline** (the artifact both compose and the EKS
  pods pull).

### Strategic positioning — what stays invariant from v0.1

- **Core knows nothing about any consumer.** The `kubernetes` source reads *standard*
  Kubernetes API resources (pods, events, nodes, services per IAM.md §A) — consumer-
  agnostic. It can observe an observed consumer cluster AND Panoptes' own cluster
  (self-monitoring). The consumer's fleet CRDs (Agones) stay in the consumer pack (v0.3).
- **Two faces, one store.** The HTTP face reads the *same* VictoriaMetrics store through
  the *same* tool functions as stdio. There is exactly one tool implementation; HTTP and
  stdio are two transports over the SAME `build_server(config)` (see § API Surface).
- **Read-only by contract.** The new `sns`/`slack` notifiers write only to *Panoptes'
  own* alert channels (an SNS topic + a Slack webhook the operator owns) — never an
  observed system. The `no_write_actions` guard must not regress (see § Authorization
  Rules). The `kubernetes` source uses read/list/watch verbs only.
- **Failure-domain independence (cluster + VPC).** The home is a *separate, dedicated*
  EKS cluster in a *dedicated VPC*, never inside an observed cluster (ARCHITECTURE §1; §
  The hosting model above). It runs in the SAME AWS account as the observed infra
  (decision #1) — so cluster + network failure domains are independent, but the AWS
  account control-plane blast radius is a deliberate, documented shared trade.
- **No anonymous MCP access.** The HTTP face is **GitHub-gated** at the nginx ingress
  (oauth2-proxy, org/team allowlist); the only unauthenticated route is a `/healthz`
  liveness probe that returns no signal data.
- **Cost discipline.** Minimal right-sized cluster (one small spot node group,
  single-AZ ok), single-node store, no Agones/HA/multi-node.

### User journey (v0.2)

1. **Operator (deploy-time, Terraform).** `terraform apply` against `modules/stack` (or
   `module "panoptes" { source = "github.com/sidkos/panoptes//modules/stack" }`),
   supplying the home region + dedicated VPC config (same account), the **GitHub OAuth
   app** client id/secret + the **org/team allowlist**, the `panoptes.<domain>` hostname,
   the GHCR image tag, the per-env read-role ARNs (possibly empty), the SNS topic ARN +
   Slack webhook. Terraform provisions the dedicated VPC + EKS cluster, the managed node
   group, the IRSA role(s), and the ingress prerequisites (nginx-ingress controller +
   cert-manager).
2. **Operator (deploy-time, Helm).** `helm install panoptes charts/panoptes -f
   values.yaml` (the Terraform outputs feed the IRSA role ARN, cluster name, ingress
   class, hostname into the values). The chart applies the workloads.
3. **Operator (config).** The hosted Panoptes config selects `dev` live + `stage`/`prod`
   `enabled:false`; `mcp.transport: http`, `auth: sso`; notifiers `sns`+`slack`; an
   `alerts:` block; a `slos:` block.
4. **LLM operator (runtime).** Any MCP client registers `https://panoptes.<domain>/mcp`,
   completes the **GitHub** SSO flow once (oauth2-proxy verifies org/team membership),
   then calls `describe_health(env="prod")`,
   `get_slo(...)`, `compare_envs(...)`, `get_cluster_state(env="dev")` — same data the
   Grafana SLO + Kubernetes dashboards show.
5. **Alerting.** The collector evaluates alert rules each cycle; firing rules go to the
   configured `sns`/`slack` notifiers.
6. **Local dev unchanged.** `docker-compose up` still runs the whole stack on a laptop
   (stdio MCP, no proxy) at zero cloud cost.

## Data Model — additions only (v0.1 canonical model unchanged)

The v0.1 `core/model.py` canonical signal model is **unchanged**. v0.2 adds:

### New derived metrics (kubernetes source)

| Derived metric | Source | Labels | Notes |
|----------------|--------|--------|-------|
| `panoptes_k8s_node_count` (gauge) | kubernetes | `env`, `cluster` | from `list nodes` |
| `panoptes_k8s_pod_restarts_total` (gauge) | kubernetes | `env`, `cluster`, `namespace` | summed container restartCounts |
| `panoptes_k8s_pods_pending` (gauge) | kubernetes | `env`, `cluster` | count of `Pending`-phase pods |
| `panoptes_k8s_pods_crashloop` (gauge) | kubernetes | `env`, `cluster` | count of `CrashLoopBackOff` containers |

`Decision: the kubernetes source emits these as derived gauges with the panoptes_ prefix
(same convention as v0.1's panoptes_health_up) because the K8s dashboard +
get_cluster_state must render from the store (parity), and pod events also surface as
live IncidentSignals (events → incident). because this keeps the "two faces, one store"
parity identical to the v0.1 sentry/cloudwatch pattern. The `cluster` label
distinguishes an observed consumer cluster from Panoptes' OWN self-observed cluster.`

### Kubernetes events → IncidentSignal

`kubernetes` `capabilities()` returns `{METRIC, INCIDENT}` (ADAPTERS §2). `Warning`-type
events normalize to `IncidentSignal` (`id`=event UID, `title`=reason+message,
`level`=`WARNING`/`FATAL`, `first_seen`/`last_seen`=event timestamps, `count`=event
`count`, `labels={env, cluster, namespace, kind}`).

> **TRACE-absence invariant (carried, re-asserted):** the union of `capabilities()`
> across all v0.2 **core** sources is exactly `{METRIC, LOG, INCIDENT}` — `kubernetes`
> adds `{METRIC, INCIDENT}`, no TRACE. The v0.1 "no trace source" negative path stays
> true. `loki`/`tempo` are v0.3.

### Alert-rule model (`core/alerts.py` — new)

| Type | Fields |
|------|--------|
| `AlertRule` (frozen dataclass) | `name: str`, `expr: str` (PromQL), `comparison: Comparison`, `threshold: float`, `for_cycles: int` (debounce, default 1), `severity: str`, `envs: list[str]` (`["all"]`=every enabled env), `labels: Labels` |
| `Comparison` (enum) | `GT="gt"`, `GE="ge"`, `LT="lt"`, `LE="le"` |
| `AlertState` (collector-held) | per-`(rule, env)` consecutive-breach counter; fires at `count >= for_cycles`, resolves on a non-breach cycle |

A firing rule produces the existing v0.1 `Alert`; `notify()` is the unchanged Notifier
contract. **No new notifier signature**; only new notifier *adapters* (`sns`/`slack`).

`Decision: alert evaluation lives in the collector (not a separate daemon) because the
collector already runs each cycle over every enabled env with a live store handle, so
rule evaluation is one extra read-query per rule per cycle — no new scheduler, no new
pod, preserving cost discipline (one collector Deployment, not a separate alert-manager
workload). The for_cycles debounce reuses the collector's existing per-source
consecutive-counter pattern.`

### SLO + MCP return types

- `SloConfig` (v0.1 — `name`+`objective`) gains an optional `query`/`window`
  (parsed-but-defaulted). `SloResult` TypedDict (`name`, `env`, `objective`, `actual`,
  `met`, `error_budget_remaining`) backs `get_slo` + the SLO dashboard.
- New `TypedDict`s in `core/mcp/`: `ClusterState` (`env`, `cluster`, `node_count`,
  `pods_pending`, `pods_crashloop`, `pod_restarts_total`, `reachable`), `EnvComparison`
  (`metric`, `window`, `per_env: dict[str, list[MetricSeries]]`, `errors: dict[str,
  str]`). Precise `TypedDict`s, no `Any`; the FastMCP/PEP-563 discipline (no
  `from __future__ import annotations` in `core/mcp/*`) applies.

## API Surface

### Plug-plane Protocols — UNCHANGED

The four v0.1 Protocols are unchanged. Every v0.2 adapter is a new class + one
`@REGISTRY.register(...)` decorator + one `core/bootstrap.py` entry.

> **`core/bootstrap.py` must gain the new core adapters** (`_CORE_ADAPTER_MODULES`):
> add `core.sources.kubernetes`, `core.notifiers.sns`, `core.notifiers.slack`.

### New source adapter (core)

| Adapter | Upstream | Provides | Mechanism |
|---------|----------|----------|-----------|
| `kubernetes` | a cluster's API server | `{METRIC, INCIDENT}` | read-only via a bound ServiceAccount token / kubeconfig context (IAM.md §A — RBAC, not IAM); `list nodes/pods/services` → gauges, `Warning` events → incidents |

`Decision: use the official kubernetes Python client over raw httpx because it handles
in-cluster vs. kubeconfig auth (Panoptes' OWN in-cluster SA token for self-observation;
a per-env kubeconfig/SA token for an observed cluster), API discovery, and pagination;
it is added to pyproject.toml dependencies. The client is wrapped behind an injectable
seam (constructor param, same pattern as the cloudwatch sts_client seam) so unit tests
stub it with a fake API object — no live cluster. because the v0.1 cloudwatch adapter
already proves the injectable-client testing pattern, and the no-async rule means the
sync client variant is used.`

> **No-write guard interaction:** the kubernetes client's read methods are
> `list_*`/`read_*`/`get_*` — none match the boto3 mutation verb set (same structural
> reason as `start_query`). No guard change is needed for the kubernetes reads.

### New notifier adapters (core)

| Adapter | Delivers to | Mechanism | Read-only-wrt-observed? |
|---------|-------------|-----------|--------------------------|
| `sns` | a Panoptes-owned SNS topic | `boto3` `sns.publish(TopicArn=…)` | YES — Panoptes' own alert channel |
| `slack` | a Slack incoming webhook | `httpx.post(webhook_url, json=…)` | YES — an alert sink, not observed |

> **CRITICAL no-write-guard note (load-bearing).** `sns.publish(...)` is in the v0.1
> no-write verb set. It is a legitimate write to Panoptes' OWN alert topic — not an
> observed system. **`Decision: place the sns notifier under core/notifiers/ and add
> sns.publish (scoped to core/notifiers/sns.py by file path) to the guard's suppression
> allowlist — NOT a blanket publish allowlist. because the guard's intent is "no writes
> to OBSERVED systems"; the alert-delivery channel is Panoptes' own. The allowlist entry
> is path-scoped + commented, so a future publish anywhere in core/sources/ still
> red-bars.`** The guard self-test gains a case asserting (a) `sns.publish(` in
> `core/sources/` is still flagged, and (b) the path-scoped suppression fires only for
> `core/notifiers/sns.py`.

### MCP server — HTTP transport (additive) + promoted tools

**The single most important API decision:** there is **one set of tool functions**
(`core/mcp/tools_discovery.py`, `core/mcp/tools_query.py`) bound via
`core/mcp/server.py::build_server`. v0.2 adds an **HTTP transport entrypoint** that
reuses the *identical* `build_server(config)` and the same `PanoptesMcpServer` /
`_FastMcpAdapter` — only the transport differs.

`Decision: add core/mcp/http.py with run_http() that calls the SAME build_server(config)
and runs the FastMCP instance over streamable HTTP (FastMCP supports
transport="streamable-http"); extend _FastMcpAdapter with run_http(host, port) alongside
its existing run() (stdio). The transport is selected by the config mcp.transport field
(stdio | http). because this guarantees "two faces, one store" at the code level — the
HTTP face cannot diverge from stdio since it is the same registered tool table; only the
FastMCP transport arg changes. The GitHub auth gate is the INGRESS's job (nginx
forward-auth → oauth2-proxy), not the server's.`

**Promoted tools (v0.1 stubs → real):**

- `get_slo(env: str, name: str) -> SloResult`.
- `compare_envs(metric: str, window: str) -> EnvComparison` (reuses the v0.1 `env="all"`

> The v0.1 `_V0_2_STUB_TOOLS` (`compare_envs`/`get_slo`/`get_cost`) flips:
> `compare_envs`/`get_slo` become real registrars; `get_cost` **stays a v0.2-stub**

**New tool (ships with the kubernetes source):**

- `get_cluster_state(env: str) -> ClusterState`.

`KNOWN_READ_ONLY_TOOLS` gains `get_slo`, `compare_envs`, `get_cluster_state` (read-shaped

## Authorization Rules

### Read into observed systems — via IRSA (the v0.2 change)

- **The home principal is now an IRSA role**, not an EC2 instance profile. The Terraform
  provisions an IAM role with an **OIDC trust to the EKS cluster's OIDC provider**, bound
  (via a `serviceAccountName` + namespace condition in the trust policy) to the
  collector and MCP **Kubernetes service accounts**. That role holds `sts:AssumeRole` for
  each configured `PanoptesReadRole/<env>` and **nothing else** (except the one
  resource-scoped `sns:Publish` below). **`Decision: IRSA over an instance profile
  because on EKS the SA-bound role is the native least-privilege path — the collector/MCP
  pods get exactly their assume-role grant via the projected SA token, no node-wide
  instance profile that every pod on the node would inherit. because a node instance
  profile would over-grant to any co-scheduled pod, weakening least privilege; IRSA scopes
  the credential to the exact SA.`** The IRSA role gives the pod its **base identity**; it
  then **assumes the per-env read-roles within the SAME account** (decision #1 — no
  cross-account trust). The per-env `PanoptesReadRole/<env>` is owned by the
  observed-side IaC (IAM.md §D); Panoptes takes the in-account ARNs as input (the v0.1
  `AWS_PROFILE` + `PANOPTES_ASSUME_ROLE_ARN` config seam keeps working unchanged).
- The `kubernetes` source uses a **read-only ServiceAccount token / kubeconfig context**
  (IAM.md §A — RBAC, `get/list/watch` on `pods, events, nodes, services`). For
  self-observation it uses Panoptes' OWN in-cluster SA (a read-only ClusterRole bound in
  the Panoptes cluster); for an observed cluster it uses a mounted per-env kubeconfig/SA
  token. EKS IAM only covers `DescribeCluster`.

> **Spec says (IAM.md edit):** IAM.md §B "The Panoptes home principal" should note that
> the home principal is now an **IRSA-bound role** on the dedicated EKS cluster (OIDC
> trust to the cluster's provider, scoped to the collector/MCP service accounts),
> replacing the instance-profile assume-role path; the grants (per-env `sts:AssumeRole`
> + the one resource-scoped `sns:Publish`) are unchanged. (Do not rewrite IAM.md in this
> planning pass — note the edit; the implementation plan's docs phase applies it.)

### MCP HTTP face — GitHub SSO at the nginx ingress (the v0.2 addition)

- The HTTP MCP endpoint is exposed through a **GitHub-gated nginx ingress**:
  `nginx.ingress.kubernetes.io/auth-url` + `auth-signin` forward-auth annotations point
  at **oauth2-proxy** configured with its **`github` provider** (org/team allowlist);
  TLS is issued by **cert-manager + Let's Encrypt**. No anonymous tool access
  (ARCHITECTURE §6, DASHBOARDS §3, ROADMAP non-negotiable). The only unauthenticated route
  is `GET /healthz` (liveness; returns `{"status":"ok"}`, no signal data) so the nginx
  ingress + the kubelet probe can check liveness without a token.
- **`Decision: GitHub OAuth2 via oauth2-proxy + nginx forward-auth (NOT ALB-native
  authenticate-oidc) because GitHub does not expose standard OIDC discovery, so the ALB
  listener's authenticate-oidc action cannot gate it; the nginx external-auth →
  oauth2-proxy(github) chain is the canonical GitHub-org gate. because the IdP choice
  (GitHub, decision #5) dictates the ingress (decision #3).`**
- **The MCP server holds no IdP secret and performs no token validation** — the
  nginx-ingress + oauth2-proxy is the trust boundary. The MCP **Service is `ClusterIP`-
  only** (never a `LoadBalancer`/`NodePort`), reachable only from inside the cluster; the
  nginx ingress is the sole public path. **`Decision: the MCP Service is ClusterIP, the
  ingress is the only public path because a LoadBalancer/NodePort MCP Service would be an
  anonymous bypass of the GitHub auth gate; ClusterIP forces every external request through
  the authenticated ingress.`** oauth2-proxy injects a verified
  `X-Auth-Request-Email`/`X-Auth-Request-User` header the server may log for audit.
- **Auditability (IAM.md §C):** oauth2-proxy logs every authenticated GitHub principal;
  the MCP server logs the forwarded identity header per tool invocation (f-string
  structured log). No PII beyond the GitHub login/email.

### Notifier write scope (the read-only-wrt-observed boundary)

- `sns`/`slack` write only to Panoptes-owned channels (`${VAR}` topic ARN + webhook URL).
  The IRSA role's IAM grants `sns:Publish` **only on that one topic ARN**
  (resource-scoped) — the single write grant in the system, on a Panoptes-owned resource.
  The Terraform makes this explicit: the `sns:Publish` statement is `Resource:
  <panoptes_alert_topic_arn>` only.

### No repo-write credential (unchanged)

New dashboards are authored as code in the consumer's repo and injected; the hosted path
adds the **`git` injection variant** (pinned commit-SHA `git fetch`, read-only deploy
plumbing — DASHBOARDS §4, IAM §C). v0.2 implements the parsed-but-deferred v0.1 `git`
selector with the SHA-pin + authenticated-transport + diff-review controls. **`Decision:
the git fetch runs at deploy time (a Helm pre-install/init job or a Terraform
null_resource), NOT in the running MCP/collector pod, because pulling code into a
long-running pod is a worse trust boundary than a reviewed deploy-time fetch (IAM §C).`**

## Migration Plan

v0.2 is **additive over a shipped v0.1**; no data migration (store schema unchanged), no
destructive code change. The migration is deployment-shape:

### Pre-migration state
- v0.1 shipped: local-compose, stdio MCP, three core dashboards/sources, `logging`
  notifier, green gate.

### Migration steps (additive, each independently revertible)
1. Add the `kubernetes` source + `get_cluster_state` + dashboard (config opt-in).
2. Add `sns`/`slack` + `alerts:` (absent = v0.1 behavior).
3. Promote `get_slo`/`compare_envs` (stub→real swap; tests updated).
4. Add the HTTP transport (`mcp.transport: http`); `stdio` stays default for local.
5. Add the Terraform EKS module + Helm chart + GHCR publish (new dirs; no existing-file
   behavior change). The dedicated cluster is stood up fresh — there is no existing host
   to migrate off (the prior single-EC2 plan was never built).

### Rollback procedure
- Each code addition is config-gated or a new file/dir; revert the config block or the
  file. The hosted layer rolls back with `helm uninstall` + `terraform destroy`. The
  v0.1 local-compose loop is never broken — it is the rollback target.

## Performance Constraints

- **Cost discipline (hard).** Minimal right-sized EKS: a single small managed node group
  (1–2 nodes, spot-eligible, single-AZ ok), single-node VictoriaMetrics (a StatefulSet
  with one PVC), no HA, no Agones, no Karpenter (decision #2 — managed node group). Honest step-up over the prior
  EC2 plan — kept minimal.
- **Alert evaluation** adds one store read-query per rule per env per cycle (O(10×envs)
  per 60s) — negligible; a failed eval is caught + logged + skipped.
- **kubernetes source** lists nodes/pods/events per cycle; bounded by the per-source
  `fetch_timeout_seconds` (default 30s), paginated; a `namespaces:` filter narrows large
  clusters.
- **HTTP MCP** adds the ingress/proxy hop; tool latency is dominated by the store query.
- **GHCR image** keeps the v0.1 layered Dockerfile cache (deps layer, then `core/`); the
  publish pipeline builds once per release tag.

## CI/CD

v0.2 **extends** the v0.1 layered GitHub Actions shape. Additions (all hermetic — no
live AWS creds, no real cluster):

### New `infra` jobs in `ci-checks.yml`

- **`terraform`:** `terraform fmt -check -recursive`, `terraform init -backend=false` +
  `terraform validate`, `tflint`. TDD-for-IaC: a plan-level assertion test
  (`terraform validate -json` / `terraform show -json` of a no-backend plan) asserts the
  IRSA role's OIDC trust + SA scoping, the resource-scoped `sns:Publish`, the
  parameterized assume-role grants, and **no wildcard/observed write action**. `Decision:
  assert on the static rendered config because a live plan needs creds CI must not hold.`
- **`helm`:** `helm lint charts/panoptes`, then `helm template charts/panoptes -f
  ci/test-values.yaml | kubeconform -strict -summary` (manifest schema validation
  against the K8s API schemas) — **offline, no cluster**. A small Python test parses the
  rendered manifests and asserts: the MCP Service is `ClusterIP` (never
  `LoadBalancer`/`NodePort`); the collector/MCP pods carry the IRSA SA annotation
  (`eks.amazonaws.com/role-arn`); the Ingress carries the **nginx GitHub forward-auth
  annotations** (`nginx.ingress.kubernetes.io/auth-url` + `auth-signin` → oauth2-proxy)
  and a cert-manager TLS annotation, with `/healthz` exempt; no workload requests a
  write-capable mount. `Decision: kubeconform over a live kubectl apply --dry-run=server
  because server-dry-run needs a cluster + creds; kubeconform validates the manifest
  schema fully offline.`

### Extended `paths-filter`
The `changes` `code` filter gains `modules/**`, `deploy/**`, `charts/**`, `Dockerfile`.
Keep `tests/**` (the v0.1 anti-rot coupling).

### Coverage floors
`core/mcp/http.py` joins `--cov=core.mcp` ≥80; `core/sources/kubernetes.py` joins
`--cov=core.sources` ≥80; `core/alerts.py` + `core/notifiers/{sns,slack}.py` are under
the global `core` ≥85. No new floor number is invented.

### New `publish` workflow — GHCR
`.github/workflows/publish.yml` on a `v0.2.*` tag: build from the existing `Dockerfile`,
tag `ghcr.io/sidkos/panoptes:<tag>`+`:latest`, push with `GITHUB_TOKEN` (`packages:write`
— the ONLY CI write permission, scoped to that job). `Decision: publish on tag only; the
Helm values + Terraform image_tag pin an immutable tag, never a moving :latest.`

### Brand grep + core-purity extension
Both guards extend their scan roots to `modules/**`, `deploy/**`, `charts/**`. Brand grep
(`grep -rin fida` → 0) now covers Terraform + Helm.
