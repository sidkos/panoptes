## How to use this playbook (for the implementer)

- **The spec is authoritative** ([`v0.3_depth_genericity.md`](v0.3_depth_genericity.md));
  this playbook adds the executable detail.
- **Prerequisite: v0.2 must be SHIPPED + green.** v0.3 extends the v0.2 hosted stack
  (kubernetes source, sns/slack, alerts, HTTP MCP, Terraform, GHCR). Do not start v0.3
  until v0.2's Definition of Done is ticked.
- **The genericity proof is the release's thesis.** The whole point is to demonstrate the
  core/consumer boundary, so every phase preserves it: core sources go in `core/`; the
  two consumer fixture packs go under `examples/` and inject via the v0.1
  `register_tools` + mounted-dir mechanism, **never touching `core/`**. The core-purity
  guard stays brand-free; fleet/Agones tokens appear ONLY under
  `examples/consumer-fleet-pack/`.
- **TDD-first.** Red tests first, implement to green, one semantic commit per phase.
- **Base gate (the v0.2 gate, by absolute venv path — the shell default python is conda
  3.11 and cannot parse PEP 695):** `.venv/bin/ruff check . && .venv/bin/ruff format
  --check . && .venv/bin/mypy --strict . && .venv/bin/pytest -m "not integration" &&
  yamllint . && <actionlint>` + both boundary guards + `grep -rin fida`→0 +
  (for IaC deltas) `terraform fmt -check && terraform validate && tflint`.

Real seams to build on (verified): `core/bootstrap.py::_CORE_ADAPTER_MODULES`,
`core/mcp/server.py` (`_V0_2_STUB_TOOLS` — `get_cost` is the last remaining stub after
v0.2; v0.3 empties it; `KNOWN_READ_ONLY_TOOLS`, `_core_registrars`), the v0.1 demo-pack
injection mechanism (`examples/demo-pack/pack.py` + the `PANOPTES_CONSUMER_PACK` hook +
`tests/unit/test_demo_pack.py` + the additive-injection invariant in
`tests/unit/test_core_purity_guard.py`), `core/sources/sentry.py`/`http_health.py`
(the httpx + respx-test pattern the prometheus/loki sources copy), `core/rest.py`
(`_format_failure`).

## 4. Risk register

| # | Risk | Resolution |
|---|------|------------|
| G1 | The fleet fixture's domain tokens could leak into `core/`. | The fleet source lives ONLY in `examples/consumer-fleet-pack/`; it `import`s core, never the reverse (the v0.1 structural import guard enforces core↛examples). Phase-4 re-runs the purity guard to prove it. |
| G2 | The two consumer packs might share a hidden core assumption (false genericity). | The signature test asserts the core registry baseline is **byte-identical** across both injections — any per-consumer core branch breaks it. |
| G3 | `get_cost` (CE) is billed/rate-limited; a per-cycle poll burns cost. | Hourly poll for the cost capability (per-source `poll_interval_seconds` default 3600); unit test asserts the cadence override is honored. |
| G4 | Shipping `tempo` would flip the TRACE-absence invariant. | `tempo` is DEFERRED (Open-Q 3); loki (LOG) only. The capabilities-union test re-asserts `{METRIC,LOG,INCIDENT}`. If the operator later adds tempo, the no-trace negative tests change in that same commit. |
| G5 | `prometheus`/`loki` httpx GETs are not boto3-shaped, so the no-write guard can't see them. | Same documented read-only known-miss as v0.1 sentry/http-health: the authoritative control is the read-only endpoint/token, not the boto3 grep. |
| G6 | The fleet source building on the core prometheus source could create a core→consumer import. | The dependency is consumer→core ONLY (the pack imports core's prometheus helpers). The structural guard catches any reverse import. |

**No open item blocks `fb`.** The four operator open-questions have recorded defaults.

## 5. Definition of Ready for `fb`

### Sufficient for autonomous implementation? **YES — once v0.2 is shipped + green.**

The spec provides the *what* (new core sources, packs, the two-consumer proof, the cost
promotion). This playbook provides the *executable how* on the proven v0.1 injection
mechanism + the v0.2 stub/registrar seams. The genericity proof is a concrete,
assertable test (byte-identical core baseline across two unrelated injections), not a
vague aspiration.

### Readiness checklist
```
[x] v0.2 is the prerequisite (must be shipped + green before Phase 0)
[x] The genericity proof is a concrete byte-identical-baseline assertion
[x] core/consumer boundary preserved (fleet tokens only under examples/; consumer→core deps only)
[x] Phase DAG + per-phase red-tests-first + exit gates + commits defined
[x] TRACE-absence invariant handled (loki ships, tempo deferred)
[x] get_cost cost-burn mitigated (hourly poll)
[x] The four operator open-questions have implementable defaults
```

### External prerequisites the OPERATOR provides (live deploy only)
1. A reachable read-only **Prometheus** endpoint — either consumer-run OR Panoptes' OWN
   dedicated EKS cluster's self-monitoring Prometheus (the `prometheus` source is agnostic
   to where it runs; it is a plain read-only HTTP target).
2. A reachable read-only **Loki** endpoint (the `loki` source's target).
3. **CE/budgets read** added to the env read-role (consumer-side).
4. (For real consumer packs) the `git`-pin trust-chain review (DASHBOARDS §4 / IAM §C).

### Verdict
**`fb`-ready once v0.2 ships.** Begin at Phase 0 and proceed phase-by-phase; the
genericity proof is the release's exit gate.
