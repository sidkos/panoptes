## How to read this spec (for `fb`)

This is the authoritative implementation contract for Panoptes **v0.3**, the stretch
release. Its thesis is **provable genericity**: prove the core is genuinely
consumer-agnostic — "not just the reference consumer's monitoring with extra steps" —
by (a) adding a core `prometheus` source + richer core dashboard packs, and (b) wiring
**two** unrelated consumer packs, each living in *its own* (illustrative) repo dir,
injected exactly as v0.1's demo pack proved.

Material design choices are recorded inline as **`Decision: … because …`** notes.
Genuinely 50/50 operator choices are in **§ Open questions for the operator**; each has
a recorded default so `fb` never stalls.

Build is **TDD-first** (unit + integration; Terraform validate/plan for any IaC). The
phase plan in the companion `v0.3_implementation_plan.md` is the commit plan. The v0.2
gate stays green and is extended, never weakened.

> **Brand-neutrality + the core/consumer boundary are the load-bearing constraint of
> this entire release.** The whole point of v0.3 is to *demonstrate* that boundary, so
> it must hold absolutely:
> - The **`prometheus`** source is **core** (consumer-agnostic — it scrapes any
>   Prometheus, reading standard metric series). It ships in `core/`.
> - The consumer's **fleet** source adapter (e.g. an Agones source scraping fleet
>   metrics *via* that Prometheus) is **consumer-pack** — it lives in the CONSUMER's
>   repo, is injected via `pack.py`, and **never enters `core/`** or the core-purity
>   guard's banned-term set. `agones`/`allocator`/`matchmaking` remain banned tokens in
>   `core/` (the guard stays brand-free; these tokens may appear ONLY under
>   `examples/` consumer-pack fixtures).
> - The **second, unrelated consumer** (a different illustrative domain) proves the core
>   is not secretly shaped around the first consumer.

## Open questions for the operator (each has a recorded default — none block `fb`)

1. **The second, unrelated consumer's domain.** v0.3 must wire a SECOND consumer pack
   that is *unrelated* to the first (game-platform) consumer, to prove genericity.
   **Default: a data-pipeline consumer** (job-lag / queue-depth / freshness packs — the
   ARCHITECTURE §3 worked example). **`Decision: a data-pipeline consumer over an
   e-commerce one because its signals (job lag, queue depth, data freshness) are
   maximally different from a game platform's fleet/matchmaking signals, so it most
   convincingly proves the core makes no domain assumption; ARCHITECTURE §3 already
   lists it as a canonical illustrative consumer.`** Both consumer packs live as
   brand-neutral fixtures under `examples/` (like the v0.1 demo pack), each in its own
   subdir, each injected via its own `pack.py`.

2. **Where the two consumer packs physically live for the genericity proof.** A real
   consumer keeps its pack in its own repo; for the in-repo *proof* we need fixtures.
   **Default: two sibling fixture packs under `examples/` —
   `examples/consumer-fleet-pack/` (the fleet/Agones-flavored one, brand-neutralized)
   and `examples/consumer-pipeline-pack/` (the second, unrelated one).** **`Decision:
   keep both as `examples/` fixtures (not new top-level dirs) because v0.1 already
   established `examples/` as the injected-fixture home the core-purity guard tolerates,
   and the genericity test injects each pack the same way the demo-pack injection test
   does — so the proof reuses the proven mechanism.`** A real consumer still points the
   injection config at its own repo dir; these are the *demonstration* fixtures.

3. **`loki`/`tempo` sources (logs/traces parity) — in or out of v0.3?** ROADMAP marks
   them **optional**. **Default: SHIP `loki` (log source) but DEFER `tempo` (traces).**
   **`Decision: ship loki because it completes log parity with a real second log source
   (alongside cloudwatch-logs) at low effort and gives the v0.1 LogSignal model a second
   producer — proving the log plane is generic too; defer tempo because no environment
   produces traces yet (ADAPTERS §2: "no tracing yet"), so a TRACE source would have no
   live target and would only flip the load-bearing TRACE-absence invariant without a
   real consumer.`** If the operator wants traces, `tempo` is a clean follow-on (it
   would change the TRACE-absence invariant — see § Data Model).

4. **Prometheus deployment for the core source's live target.** The `prometheus` source
   needs a Prometheus to scrape. **Default: a Prometheus that ALREADY runs somewhere
   reachable — either in an observed CONSUMER cluster OR in Panoptes' OWN dedicated EKS
   cluster (self-observation) — scraped read-only over HTTP `/api/v1/query[_range]`.**
   **`Decision: the source reads from an existing Prometheus (consumer-run or Panoptes'
   own in-cluster one), NOT a Panoptes-provisioned-for-the-consumer Prometheus, because
   Panoptes' cost discipline says single-node store + no extra managed infra. Since v0.2
   Panoptes runs on its own EKS cluster, a lightweight Prometheus MAY already live there
   for self-monitoring; the source treats it as just another read-only scrape target
   (distinguished by the env/cluster label), exactly like a consumer's Prometheus.`** The
   source is agnostic to WHERE the Prometheus runs — it is a plain read-only HTTP target.

## Product Specification

### Overview

v0.3 deepens the core and *proves* its genericity:

- A core **`prometheus`** source (read-only PromQL scrape of any reachable Prometheus).
- **Richer core dashboard packs**: Kubernetes/Karpenter (extends v0.2's k8s pack), Cost
  (with the now-real `get_cost` + the CE/budgets read grant), Datastore.
- **Two unrelated consumer packs** wired in as injected fixtures to prove the core is
  generic — including the **fleet** consumer-pack *source adapter* (domain-specific,
  consumer-owned, brand-neutralized in the fixture) driving a Fleet dashboard.
- Optional **`loki`** log source (shipped by default per Open-Q 3); **`tempo`** deferred.

### Strategic positioning — the genericity proof (the whole point)

The release succeeds iff, at the end:

1. `core/` contains the `prometheus`/`loki` sources + richer core packs and **zero**
   consumer-domain tokens (the core-purity guard, unchanged-and-brand-free, is green).
2. **Two unrelated consumer packs** (fleet + pipeline) each inject cleanly via the v0.1
   `PANOPTES_CONSUMER_PACK` + mounted-dir / `git` mechanism, each registering its own
   source adapter + MCP tools + dashboards, **without a single core change between
   them**. The additive-injection runtime invariant (v0.1 guard) holds for each.
3. The fleet consumer's source adapter (e.g. Agones-over-Prometheus) lives entirely in
   `examples/consumer-fleet-pack/pack.py` — it `import`s the core `prometheus` source's
   normalization helpers (or wraps the core source), proving a consumer pack can BUILD
   ON a core source while staying out of core.

`Decision: the genericity proof is "two unrelated consumers, zero core diff between
them" because that is the only test that distinguishes a genuinely generic core from one
secretly shaped around consumer #1. The fleet pack consuming the core prometheus source
proves the consumer→core dependency direction is correct (consumer depends on core,
never the reverse — the v0.1 structural import guard already enforces core↛examples).`

### User journey (v0.3)

1. **Consumer A (game platform).** Points its injection config at
   `examples/consumer-fleet-pack/` (the brand-neutral fleet fixture): a `prometheus`-
   backed fleet source adapter + a `get_fleet_health` MCP tool + a Fleet dashboard.
2. **Consumer B (data pipeline).** Points a SECOND injection config at
   `examples/consumer-pipeline-pack/`: a job-lag/queue-depth source + a
   `get_pipeline_lag` tool + a freshness dashboard. **No core change** distinguishes the
   two deploys — only the injected pack dir.
3. **Operator.** Adds the core `prometheus`/`loki` sources + the richer core packs
   (Karpenter, Cost, Datastore) to config; `get_cost` is now real with the CE/budgets
   read grant.
4. **LLM operator.** Calls core tools (`get_cost`, `query_metric` over Prometheus
   series) AND each consumer's injected tools (`get_fleet_health`, `get_pipeline_lag`)
   over the same hosted SSO-gated HTTP MCP — same data the dashboards show.

## Data Model — additions only

The canonical model is unchanged. v0.3 adds derived/native metrics:

| Derived/native metric | Source | Notes |
|-----------------------|--------|-------|
| (passthrough Prometheus series) | prometheus (core) | the source scrapes native Prometheus series and normalizes to `MetricSignal` with `env` stamped |
| `panoptes_cost_spend` (gauge) | cloudwatch (CE/budgets) | per-service spend; backs the Cost dashboard + `get_cost` |
| `panoptes_cost_budget_burn` (gauge) | cloudwatch (CE/budgets) | budget burn fraction |
| (Loki log lines → `LogSignal`) | loki (core, default-shipped) | second log producer alongside cloudwatch-logs |
| (consumer fleet metrics) | **consumer-pack** fleet source | e.g. `ready`/`allocated`/`reserved` — lives in `examples/consumer-fleet-pack/`, NOT core |

> **TRACE-absence invariant (still holds — load-bearing).** With `loki` (LOG) added and
> `tempo` DEFERRED (Open-Q 3), the union of `capabilities()` across all core sources is
> still exactly `{METRIC, LOG, INCIDENT}` — no TRACE. The v0.1/v0.2 "no trace source"
> negative path stays true. **If the operator later ships `tempo`, this invariant flips
> to include TRACE** and the no-trace negative tests must be updated in that same change
> (called out here so it is not a surprise).

### Cost types (promote `get_cost` from stub to real)

| TypedDict | Returned by | Fields |
|-----------|-------------|--------|
| `CostBreakdown` | `get_cost(env, window)` | `env`, `window`, `total: float`, `per_service: dict[str, float]`, `budget_burn: float` |

`get_cost` moves out of `_V0_2_STUB_TOOLS` (it was the last remaining stub). The
home-principal IAM (v0.2 Terraform) gains the `ce:GetCostAndUsage` + `budgets:ViewBudget`
read grants for the env's read-role (IAM.md §A "Cost dashboard (v0.2+)" row — these are
**read** actions, no write).

## API Surface

### Plug-plane Protocols — UNCHANGED

All four v0.1 Protocols unchanged. Every v0.3 adapter is a new class + a registry
decorator + a `core/bootstrap.py` entry (for core sources) OR a consumer `pack.py`
registration (for the two consumer packs — never touching `core/bootstrap.py`).

### New core source adapters

| Adapter | Upstream | Provides | Mechanism |
|---------|----------|----------|-----------|
| `prometheus` | any reachable Prometheus | `{METRIC}` | `httpx` GET `/api/v1/query` + `/api/v1/query_range` (read-only); normalizes series → `MetricSignal` with `env` |
| `loki` (default-shipped per Open-Q 3) | a Loki instance | `{LOG}` | `httpx` GET `/loki/api/v1/query_range` (read-only) → `LogSignal` |

> Add both to `core/bootstrap.py::_CORE_ADAPTER_MODULES`. Both are read-only (httpx GET);
> the no-write guard (boto3-shaped) does not flag them, and their httpx calls are GET —
> documented read-only known-miss surface (the IAM/read-token discipline is the
> authoritative control, same as v0.1 sentry/http-health).

### Promoted tool

- `get_cost(env: str, window: str) -> CostBreakdown` — out of `_V0_2_STUB_TOOLS`; reads
  `panoptes_cost_*` gauges. `KNOWN_READ_ONLY_TOOLS` gains `get_cost`. After this,
  `_V0_2_STUB_TOOLS` is **empty** — and the structural read-only test asserts the
  default-config tool set is the complete v0.3 core set, with no remaining stub.

### Consumer-pack tools (NOT core — injected via two fixture packs)

| Tool | Pack | Notes |
|------|------|-------|
| `get_fleet_health(env)` | `examples/consumer-fleet-pack/` | reads fleet metrics the fleet source scraped from the consumer's Prometheus; registered by the pack's `register_tools` |
| `get_pipeline_lag(env)` | `examples/consumer-pipeline-pack/` | reads job-lag/queue-depth; the second, unrelated consumer's tool |

These register exactly via the v0.1 `register_tools(mcp_server)` hook — `core/` never
imports them.

### Lambda touchpoints

**Not applicable** (Panoptes has no Lambda).

## Authorization Rules

- **`prometheus`/`loki` sources** are read-only HTTP scrapes (GET); their endpoints are
  reachable read-only targets the home host can reach. No new write surface.
- **`get_cost`** requires the env read-role to grant `ce:GetCostAndUsage` +
  `budgets:ViewBudget` (read-only, IAM.md §A). The v0.2 Terraform's per-env assume-role
  grant is unchanged (Panoptes assumes the consumer's read-role); the **consumer** adds
  these two read actions to its read-role policy (consumer-side, IAM.md §D).
- **Consumer-pack `pack.py` is executed in the collector/MCP process** — the v0.2 `git`
  injection trust chain (full-SHA pin + authenticated transport + diff-review gate,
  DASHBOARDS §4 / IAM §C) applies to BOTH consumer packs. The genericity proof's two
  packs are in-repo fixtures (no remote fetch), so they exercise the mounted-dir path;
  the doc must reiterate the `git`-path trust gate for real remote consumer packs.
- **The fleet consumer source** still reads only (scrapes the consumer's Prometheus);
  no write to any observed system. The consumer's Kubernetes read-role may add
  `agones.dev` CRD read (IAM.md §A "consumer example only" — consumer-side, not core).

## Performance Constraints

- `prometheus`/`loki` scrapes are bounded by the per-source `fetch_timeout_seconds`
  (v0.1 default 30s) and paged; the source must bound the query window.
- `get_cost` (CE) is **rate-limited + billed per request** — `Decision: cache the cost
  query to once per N cycles (default hourly), not every 60s cycle, because Cost
  Explorer charges per API call and updates at most daily; a per-minute poll would burn
  cost to monitor cost. The source carries a `poll_interval_seconds` override (the v0.1
  per-source timing field already supports this) defaulting to 3600 for the cost
  capability.`
- Two consumer packs do not change core performance — each is an injected addition.

Extends the v0.2 gate. Additions:

- `core/sources/prometheus.py` + `core/sources/loki.py` join the `--cov=core.sources`

  domain tokens (`agones`/fleet terms) live ONLY under
  `examples/consumer-fleet-pack/` — the guard scans `core/` only for those tokens, so
  the fixture is tolerated exactly as v0.1's demo pack is.

      `capabilities()=={METRIC}`, unit-tested (respx, no live Prometheus).
- [ ] `loki` core source scrapes read-only → `LogSignal`, `capabilities()=={LOG}`;
      `tempo` explicitly NOT shipped; the TRACE-absence invariant

      (consumer #1, builds on the core `prometheus` source, fleet tokens ONLY under
