## How to use this playbook (for the implementer)

- **The spec is authoritative.** This playbook references the spec's sections by
  name and adds the *executable detail* needed to build without stalling. Where
  this playbook and the spec disagree, the spec wins — open an issue, do not
  silently diverge.
- **TDD-first across all applicable layers.** Each phase below lists the test
  files written **red first**, then the implementation files that turn them
  green. The order inside each phase is dependency order: create top-to-bottom.
- **One semantic commit per phase**, exactly as the spec's `## Rollout Phases`
  prescribes. Each phase ends with an explicit exit-criteria checklist (below)
  that must be green before the commit and before the next phase starts.
- **No reference-consumer changes.** This work is entirely inside the `panoptes`
  repo. The reference consumer (a game platform) is only a live `dev` target for
  CloudWatch + Sentry + `/health`. Keep the repo brand-neutral: no consumer brand
  name appears anywhere, including the boundary guard — which is reconceived to be
  brand-free (a structural "`core/` must not import `examples/`" check plus generic
  consumer-domain terms; see below).
- **Gate per phase:** `ruff check . && ruff format --check . && mypy --strict . && pytest -m "not integration"` must pass before each commit. The integration
  marker suite runs only in Phase 8 (and the CI integration job).

## 1. Inter-phase dependency graph & ordering rationale

```
Phase 0  scaffold + CI            ──┐  (nothing green required; bootstraps the gate)
                                    ▼
Phase 1  model + registry + config ──┐  (pure core, no I/O; everything below imports it)
                                     ▼
            ┌────────────────────────┼────────────────────────┐
            ▼                        ▼                         ▼
Phase 2  stores            Phase 3  sources           (model/registry only)
   (Store Protocol)          (Source Protocol;
            │                 derived metrics)
            └───────────┬────────────┘
                        ▼
Phase 4  collector + notifier   (needs sources to fetch + a store to write)
                        │
                        ▼
Phase 5  grafana provider + core packs + injection glob
                        │     (needs nothing from collector; needs config+model;
                        │      placed here so dashboards exist before MCP reads them)
                        ▼
Phase 6  MCP stdio + core tools (reads store + sources + dashboard catalog)
                        │
                        ▼
Phase 7  demo consumer pack     (proves injection: registers adapter + get_demo_signal;
                        │        core-purity guard must already exist from Phase 0)
                        ▼
Phase 8  docker-compose + integration suite (wires every prior phase end-to-end)
                        │
                        ▼
Phase 9  docs & polish
```

**Hard ordering rules (what must be GREEN before a phase starts):**

| Phase | Cannot start until | Why |
|-------|--------------------|-----|
| 1 | Phase 0 CI green on skeleton | the gate must exist before any real code lands |
| 2 | Phase 1 green | stores consume `CanonicalSignal`, `MetricQuery`, `MetricSeries`, `Store` Protocol, registry |
| 3 | Phase 1 green | sources emit `CanonicalSignal` + derived metrics; need `Source` Protocol + errors |
| 4 | Phases 2 **and** 3 green | collector wires a real source set to a real store; notifier needs `Alert` |
| 5 | Phase 1 green | provider needs `DashboardPack`, `DashboardsConfig`, `ConsumerPackConfig`; independent of 2–4 |
| 6 | Phases 2, 3, 5 green | MCP tools read the store (2), live sources (3), and the dashboard catalog (5) |
| 7 | Phases 5 **and** 6 green; Phase 0 guard present | demo pack registers an adapter (registry, P1) + an MCP tool (P6) + a dashboard the P5 glob picks up; purity guard must be live to assert no leak |
| 8 | Phases 2–7 green | compose brings up the whole stack; integration tests exercise collector→VM→MCP→Grafana |
| 9 | Phase 8 green | docs describe the working stack |

Phases 2 and 3 are independent of each other and **may be built in either order**
(both depend only on Phase 1). Phase 5 is independent of 2–4. Everything else is
strictly serial.

### 3.3 The two boundary-guard tests (exact behavior)

- `tests/unit/test_core_purity_guard.py`
  - **Primary control — structural import check:** walk `core/**/*.py`; flag any
    `from examples` / `import examples` (and assert no `core/` module resolves a
    path inside `examples/`). This is the real anti-coupling guarantee.
  - **Defense in depth — generic-term grep:** `BANNED = frozenset({"allocator", "matchmaking", "agones"})`
    (generic consumer-domain terms; **no consumer brand literal**). For each
    `core/**/*.py`, lowercase + word-boundary regex per banned token; assert empty
    match list naming file + token. `demo`/`game` are intentionally excluded —
    `demo` is the example pack's own name and `game` is a common English substring;
    banning them adds no coupling protection beyond the structural check and is
    false-positive prone.
  - **The guard is brand-free** — it contains no consumer brand string. (If a real
    consumer ever needs a brand tripwire in *its own* pack, it would byte-encode
    the token rather than embedding plaintext; core never needs one.)
- `tests/unit/test_no_write_actions_guard.py`
  - walk `core/**/*.py` and `examples/**/*.py`. **Snake_case boto3 mutation shape
    (genuine-AWS-mutation verbs, call-shape anchored):** match the method-call
    shape `\.[a-z_]*(put_|create_|update_|delete_|write_|upload_|copy_|publish|send_message|send_raw_email|invoke|terminate_|reboot_|attach_|detach_|enable_|disable_|untag_|associate_|disassociate_|deregister_|batch_write|transact_write|modify_|set_alarm|start_instances|stop_instances|stop_db_instance|put_object|put_item)[a-z_]*\(`.
    The trailing `(` is **required** (so plain identifiers, attribute names, and
    prose like a `set_window` variable or an `update_` comment do not
    false-positive — the open paren follows the method name on the same token in
    idiomatic boto3, and the regex still spans kwargs/multi-line argument lists).
    The `[a-z_]*` framing catches mid-name forms like `batch_write_item` /
    `transact_write_items`. **The verb set deliberately OMITS the bare
    `register_`/`add_`/`set_`/`run_`/`stop_`/`tag_`/`remove_` families** — they
    would collide with Panoptes' own `register_tools`/`add_tool`/`add_source`
    methods and with read-only `stop_query`/`start_query` (below). **Secondary
    PascalCase shape:** `r"\b(Put|Create|Update|Delete|Write)[A-Z]\w*"` with a
    known-safe allowlist (`UpdatedAt`, `CreatedAt`, `DeleteMarker`, `Writer`) to
    avoid false positives on read-response field/type names — or drop this shape
    entirely and rely on the snake_case set (boto3 Python call sites are
    snake_case anyway); `fb` picks one and documents the choice.
  - **Suppression allowlist — only the genuinely-matched file-I/O verbs (stated
    explicitly so the guard is deterministic):**
    - **file I/O** — `write_text`, `write_bytes` (these contain `write_` and **do**
      match the verb set; the allowlist suppresses them — needed because the
      Phase-5 Grafana provider writes JSON to disk, Risk R6). These are the **only**
      members of the suppression allowlist. `writelines` and `writer` need **no**
      entry: there is no `write` alternative lacking a trailing `_`, so they never
      match in the first place.
  - **Never-matched by construction (NOT suppression-allowlist members — the verb
    set omits their bare families, so they produce no hit to suppress):**
    - **read-only CloudWatch Logs Insights** — `start_query`, `stop_query`,
      `get_query_results`. The verb set has no bare `start_`/`stop_`/`get_` family,
      so these never match. IAM.md §A grants `logs:StartQuery`/`logs:StopQuery`/
      `logs:GetQueryResults` as **read-only** actions, so a future Logs-Insights
      cloudwatch adapter calling `client.stop_query(...)` is never a false positive.
    - **Panoptes registration helpers** — `register_tools`, `register`, `add_tool`,
      `add_source`. The verb set has no bare `register_`/`add_` family, so these
      never match. They are mandated by the spec (`core/mcp/server.py` defines and
      calls `register_tools(mcp_server)`; the demo pack's `pack.py` defines
      `register_tools`) and are not boto3 mutations. (They MAY appear in the
      allowlist as documentation of intent, but the allowlist is a no-op for them.)
  - **Self-test (regression-safe matcher):** feed the matcher synthetic known-bad
    strings (`ddb.put_item(`, `s3.upload_file(`, `sns.publish(`,
    `dynamodb.batch_write_item(`, `ec2.terminate_instances(`, `client.create_table(`)
    and assert **all** are flagged. Then prove the non-flagged forms for the right
    reason, **matching the buckets to regex reality** (the verb set omits the bare
    `register_`/`add_`/`start_`/`stop_`/`get_` families):
    - **matched-then-suppressed** (the ONLY members — they contain a matched
      `write_` verb and would match without the allowlist): `path.write_text(`,
      `path.write_bytes(` → assert they match the raw pattern AND are suppressed by
      the allowlist;
    - **never-matched** (assert `pattern.search(...) is None`): `client.stop_query(`,
      `client.start_query(`, `client.get_query_results(`, `mcp.register_tools(`,
      `pack.add_tool(`, `provider.add_source(`, `f.writelines(`, `csv.writer(` —
      the verb set has no alternative that matches them, so they produce **no match
      at all** (not a vacuous allowlist hit).
    This proves both the matcher shape AND that the verb set is built so it never
    collides with Panoptes' own `register_tools`/`add_tool` symbols or the
    read-only Logs actions (they are never matched, not suppressed).
  - **Note:** this grep is heuristic defense-in-depth and is **boto3-shaped only**
    — it does NOT inspect raw `httpx` POST/PUT/DELETE bodies (the `sentry` /
    `http-health` sources use httpx, not boto3). **Documented known-miss shapes**
    (explicit blind spots, not silent): paginator-wrapped mutations; dynamic
    `getattr(client, verb)(...)` dispatch where the verb is not a literal token; the
    low-level `client.api_call(...)` / botocore `make_api_call` escape hatch; and
    any new AWS mutation verb absent from the curated set. The authoritative
    no-write controls are the **read-only IAM role** (IAM.md hard rule) plus the
    **read/viewer-scoped upstream tokens** for httpx-based sources.

Without the file-I/O suppression, Phase 5 cannot go green; the verb-set omission keeps Phases 6-8 (which define/call `register_tools`/`add_tool`) green without relying on a suppression entry. |
| R7 | CI `changes` filter sets `run-integration: true` on first push (code paths exist), so the integration job runs from Phase 0 before any integration test exists. | **Resolved IN the canonical YAML:** the spec's `## CI/CD` integration job is `pytest -m integration \|\| [ $? -eq 5 ]`, tolerating exit code 5 ("no tests collected") through Phases 0-7. "Byte-for-byte the spec YAML" therefore goes green on the skeleton — no contradiction. **Phase 8 both removes the `\|\| [ $? -eq 5 ]` tolerance AND asserts `pytest -m integration` collects ≥ 5 tests** once real integration tests land — keyed to assertion count (collector-to-VM, grafana-provisioning, and the three separate `test_mcp_stdio_e2e` functions), so neither an empty collection nor a quiet file-consolidation can silently green the gate (spec Phase 8 exit criterion). |
| R8 | `DemoSignal`/`SignalCatalog`/`DashboardData`/`HealthRollup`/`DashboardSummary` TypedDict fields not fully enumerated in the spec. | `fb` defines them as precise `TypedDict`s in the owning module — discovery/query types in `core/mcp/` (P6), `DemoSignal` in `examples/demo-pack/pack.py` (P7). Fields follow the spec prose (e.g. `HealthRollup` = per-source reachability + key derived metrics + open-incident count). No `Any`; nested TypedDicts allowed (mind PEP-563). |
| R9 | VictoriaMetrics write format (JSON-line import vs Prometheus remote-write). | **Use the VM JSON-line import endpoint** (`/api/v1/import`) — simplest to assert byte-exactly in unit tests; PromQL read-back via `/api/v1/query_range`. |
| R10 | Live AWS profile + Sentry token are needed for a *real* `docker-compose up`, but CI must be hermetic. | Integration tests (CI + local `pytest -m integration`) use **synthetic `panoptes_*` series only** — no live AWS/Sentry. Live-credential runs are the manual operator journey, gated on the operator providing `.env`. CI needs **no** cloud secrets. |
| R11 | Collector scheduled (60s) vs `--once`. | Spec Open Q1 locked: 60s default, per-source `poll_interval_seconds` override, `--once` for tests/CI. Tests use `--once`; the loop path gets a focused unit test (one iteration via a monkeypatched sleep/break). |
| R12 | FastMCP decorator-based tool registration may not type-check cleanly under `mypy --strict` (`disallow_any_explicit`/`disallow_any_generics`; untyped decorator loses precise type). | The FastMCP tool-registration decorator may need `# type: ignore[misc]` on the decorator line (or a typed wrapper). **Apply the ignore ONLY if mypy actually reports the error — do NOT add it speculatively:** `--strict` implies `warn_unused_ignores`, so an unneeded `# type: ignore` on a FastMCP version that ships clean stubs (`py.typed`) makes mypy `--strict` FAIL on the unused ignore — the opposite failure mode. `fb` verifies mypy is clean against the pinned `fastmcp` version both with and without the ignore, and prefers a typed wrapper function (no per-call ignore) where it removes the seam entirely. This is the **one permitted strict-mypy escape**; a decorator typing failure is expected, not a redesign trigger. FastMCP is already in the `ignore_missing_imports` override (§3.1). |
| R13 | Grafana file-provisioning + VM readiness are async after container start; an immediate integration assertion races them. | The Grafana integration test **polls `GET /api/search`** (bounded retry/backoff) until all four dashboards appear; `test_collector_to_vm` polls VM `/health` (or `/api/v1/query` → 200) before the PromQL read-back. No single immediate assertion. (Spec Phase 8 / `## Tests` Integration.) |
| R14 | The Phase-6 `get_dashboard_data` unit test must parse `panel.targets[].expr`, but the shipped `core/dashboards` JSON only exists from Phase 5. | The Phase-6 unit test uses a small **inline/`tmp_path` dashboard JSON fixture** with known `panel.targets[].expr` — self-contained, not dependent on Phase-5 artifacts. The shipped core JSON is exercised by `test_dashboards_valid` (P5) and the Phase-8 integration test. |
| R15 | The Grafana↔MCP parity integration test could flake on VictoriaMetrics staleness/last-value carry-forward (a single sample repeated across interior `/query_range` grid points) if it compares the full rounded-tuple SET. | **Pin both callers to the same `/query_range` endpoint family** (identical `start`/`end`/`step`, never vs an instant `/query`); make the **scalar last-value at the sample's own grid point the PRIMARY assertion** (robust against carry-forward). The rounded-tuple comparison is optional/secondary and compared at the sample's own grid point, not the full grid (spec Phase 8 / `## Tests` Integration). |
